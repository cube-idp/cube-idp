## Proposal: Kernel+Packs (neocube)

**Philosophy:** neocube is a pusher, not an operator. idpbuilder's deepest complexity — an in-process controller-runtime manager, three CRDs, a git server with an admin user, and a cnoe:// URL-rewriting scheme — exists to simulate GitOps for local dirs. neocube deletes all of it: the kernel is a single static Go binary that (1) ensures a cluster exists via a compiled-in provider, (2) server-side-applies a minimal Flux + in-cluster OCI registry, and (3) exits. Everything else — ingress, Argo CD, Gitea, Backstage, secrets tooling — is a data-only "pack": versioned files fetched from OCI or git, rendered in-process, reconciled by Flux. There are no RPC plugins, no Wasm runtime, no daemons; extensibility is data first, exec second, code-in-tree last. The mental model fits in one sentence — "neocube puts your platform config where Flux can see it" — and the happy path (`neocube up`) is cluster + GitOps in under 60 seconds with nothing else installed.

**Components:**
- **Command shell** (spf13/cobra v1.10.x for commands; charm.land/huh v2 for the rare prompt; charm.land/lipgloss v2 for status lines (no full Bubble Tea app in the kernel — a progress spinner is not a TUI); log/slog with --debug for raw controller logs.): Cobra command tree (up/down/status/sync/pack/trust/doctor), interactive prompts only when a required value is missing, single-line progress rendering per phase with a hard timeout and a failure explainer (fixes idpbuilder #430/#17/#434 'hangs forever with no diagnostics').
- **Cluster providers (compiled-in, tiny interface)** (sigs.k8s.io/kind/pkg/cluster v0.32 (library API, auto-detects docker/podman/nerdctl); github.com/k3d-io/k3d/v5 v5.9 typed config; k8s.io/client-go for the 'existing' provider (kubeconfig context selection, honoring KUBECONFIG — fixes #362).): One 4-method Go interface: Ensure(ctx, spec) (kubeconfig, error), Delete, Exists, Diagnose. Exactly three built-in implementations at v1: kind (default), k3d, and 'existing' (any kubeconfig context — this alone closes idpbuilder #74/#246/#530). Providers are compiled in, not plugins: a new provider is a PR, because the interface is small enough that RPC indirection costs more than it buys.
- **Apply engine (the anti-controller)** (github.com/fluxcd/pkg/ssa ResourceManager (SSA apply, ordered stages, WaitForSet with kstatus health checks, inventory-based pruning) — the exact library Flux built to replace kubectl; k8s.io/cli-runtime only for kubeconfig plumbing.): The single biggest structural change vs idpbuilder: neocube runs NO controller-runtime manager and installs NO neocube CRDs. `neocube up` is a one-shot, idempotent server-side-apply with per-resource health waits, an inventory record in a ConfigMap, and prune-on-next-run. Re-running `up` is the upgrade story; `down` reads the inventory for cascading delete and removes the kube context (fixes #351/#458/#489/#440 and the missing upgrade path in one mechanism).
- **GitOps core: Flux + in-cluster OCI registry (no git server)** (Flux v2.9 controller manifests embedded at build time via go:embed (source-controller, kustomize-controller, helm-controller only — no notification/image controllers); github.com/fluxcd/pkg/oci + oras-go v2.6 for artifact push; project-zot/zot as the embedded registry manifest (single binary, OCI-native, ~0 config).): The only thing installed by a bare `neocube up` besides the cluster: Flux source/kustomize/helm controllers plus a 20MB in-cluster OCI registry. Local directories are delivered as Flux OCIRepository artifacts pushed to that registry — which deletes idpbuilder's entire Gitea axis: no giteaAdmin, no password drift (#296/#304), no cnoe:// URL rewriting, no split-horizon git URLs (#300/#398). Gitea and Argo CD still exist — as optional packs for people who want a git UI or the Argo UI.
- **Pack engine (data-only packages)** (helm.sh/helm/v4 SDK (action package, SSA-by-default installs, no helm binary); sigs.k8s.io/kustomize/api for overlay ingestion; cuelang.org/go v0.17 (pure Go, zero deps) for pack schemas and typed values; go-git + oras-go for fetch; sha256/commit pins recorded in cube.lock.): Loads packs from local dir, git ref (commit-pinned), or OCI reference; validates pack.cue metadata + user values against the pack's CUE schema; renders Helm charts in-process and kustomizations; hands the object set to the apply engine or, for gitops-managed packs, publishes them as OCI artifacts with a Flux Kustomization/HelmRelease. A pack is files, never code — the Devbox lesson: ~80% of platform extensions are manifests + chart refs + values + a service definition.
- **Trust & networking helper** (crypto/x509 + github.com/smallstep/truststore (the mkcert mechanism, as a library) for OS trust; containerd certs.d files mounted via the kind provider; CoreDNS Corefile patch applied through the same SSA engine.): Generates a local CA once per machine (~/.neocube/ca), issues the wildcard cert, and — with explicit consent via `neocube trust` — installs the CA into the OS/browser trust stores AND into the cluster nodes' containerd config, killing the largest open pain cluster in idpbuilder (#451/#545/#389/#534/#520: docker push, in-cluster pulls, Backstage↔git trust). BYO cert is a config field. CoreDNS rewrite makes one canonical hostname (*.cube.localtest.me or sslip.io) resolve identically inside and outside the cluster.
- **Config loader** (cuelang.org/go for schema definition + validation of the YAML (CUE natively validates YAML/JSON); JSON Schema exported at build time and served from the repo for yaml-language-server.): One file, cube.yaml, apiVersion neocube.dev/v1 kind Cube, with a published JSON Schema (generated from the CUE definition) for editor autocomplete, strict-by-default validation with precise error positions, and explicit config-version migration (`neocube migrate`) — k3d's model, with the lesson learned: commit to v1 semantics at launch instead of years of v1alphaN churn.
- **Doctor / preflight** (Docker Engine API socket probing (DOCKER_HOST/podman socket, the testcontainers-go pattern); kstatus conditions + core/v1 Events via client-go for the failure explainer.): Runs before every `up` and on demand: container runtime reachable, cgroup pid limits, ports free, disk, rate-limit-prone registries mirrored, corporate MITM proxy detection. Every wait failure in the apply engine ends with a rendered diagnosis (pod events, image pull errors, last container log lines) instead of an infinite spinner.

**Extensibility:** Three tiers, deliberately ordered by decreasing frequency and increasing cost, with no RPC-plugin tier at all. Tier 1 — Packs (the primary mechanism, covers ~80–90% of extensions): a pack is a directory containing pack.cue (name, version, dependencies, a CUE #Values schema) plus plain Kubernetes manifests, Flux HelmRelease/Kustomization objects, or a chart reference; zero executable content. Packs are addressed as `./local/dir`, `github.com/org/repo//path@v1.2.3` (commit pin recorded in cube.lock, Dagger's lesson), or `oci://ghcr.io/org/pack:v1` pushed/pulled with oras-go (Crossplane xpkg's lesson: reuse OCI registries, never run a bespoke one). The neocube org ships starter packs — traefik (nginx is EOL), argocd, gitea, backstage, cert-manager, external-secrets — and a `cnoe-compat` pack loader that ingests existing CNOE stacks' Argo Application YAMLs, translating cnoe:// paths into OCI artifact pushes, to bootstrap from idpbuilder's ecosystem. Tier 2 — exec subcommands (krew model): any `neocube-<name>` binary on PATH becomes `neocube <name>`, receiving kubeconfig path, cube.yaml path, and inventory name via env vars; distribution is a git-repo index of sha256-pinned archives with an explicit unaudited-plugin warning. Tier 3 — compiled-in Go interfaces: ClusterProvider and PackSource are small interfaces in pkg/, and new implementations land as PRs; we explicitly reject hashicorp/go-plugin here because two-to-four providers do not justify protocol versioning, handshake code, and per-OS plugin binaries. Extism/Wasm hooks (validators/transformers, Helm 4's HIP-0026 pattern) are on the watchlist and will be adopted only after Helm 4's system proves out — and only as a Tier-1.5 for packs that genuinely need logic, never for core.

**UX:**

$ neocube up --context my-eks                  # same platform onto an EXISTING cluster, no kind
$ neocube up -p oci://ghcr.io/neocube/packs/dev-stack:v1   # ad-hoc pack, no config file needed
$ neocube trust                                # one-time: install local CA into OS + cluster trust
$ neocube sync ./platform --watch              # push dir as OCI artifact on change; Flux reconciles (replaces --no-exit + Gitea push loop)
$ neocube status                               # kstatus health of every inventory resource
$ neocube get secrets argocd                   # credentials, same ergonomics as idpbuilder
$ neocube vendor                               # pull all pack/chart/image refs into ./vendor for air-gap
$ neocube down                                 # inventory-driven cascading delete + kube-context cleanup

**Differentiators vs idpbuilder:**
- Works on existing/remote clusters via --context, not just kind — directly closes idpbuilder #74/#246/#530, its oldest and most-deferred architectural gap.
- No in-process controller-runtime, no CRDs, no daemon: `up` is one-shot idempotent SSA with inventory; re-running `up` IS the upgrade command and `down` IS cascading delete — the lifecycle gaps (#351/#458/#489, no upgrade story) are solved structurally, not patched.
- Gitea and the cnoe:// scheme are gone from the core: local dirs ship as Flux OCIRepository artifacts to a tiny in-cluster zot registry, eliminating giteaAdmin literals, password drift, and split-horizon git URLs (#296/#300/#304/#398); Gitea remains available as a pack.
- Certificate pain solved at the OS level: mkcert-style local CA installed into system trust stores AND node containerd trust with one consented command (`neocube trust`), plus BYO certs — versus idpbuilder's five open cert workaround issues (#451/#545/#389/#555/#534).
- Nothing is compiled in but Flux: ingress controller, GitOps UI (Argo), git server, everything is a versioned OCI pack — the ingress-nginx EOL fire-drill (#552/#565) becomes `packs: traefik:v1` instead of a code fork.
- Every wait has a timeout and ends in a diagnosis (kstatus + pod events + image-pull errors) instead of an infinite 'Waiting for Deployment...' spinner — the #430 class of failure (57 comments) becomes an error message.
- First-class air-gap: `neocube vendor` pins and localizes every pack/chart/image reference; registry mirrors are config, not flags (#402/#501/#229).
- Typed, schema-versioned config (CUE-validated cube.yaml + published JSON Schema) replaces 17 create-flags and a nonstandard #{} template dialect.
- Argo-ecosystem on-ramp preserved: the cnoe-compat loader ingests existing CNOE stacks (-p Argo Application YAMLs), so migration is additive rather than a rewrite of community content.

**Tradeoffs:**
- Flux as the built-in engine vs Argo CD's mindshare: Argo is not embeddable as a library (k8s.io/kubernetes replace directives, issue #4055), so making it optional-but-installed is architecturally correct — yet CNOE's audience thinks in Argo Applications, and the compat loader will never cover 100% of ApplicationSet generator exotica.
- No resident controller means no continuous drift correction from the CLI side between runs; we accept 'reconcile happens in-cluster (Flux) or at next `up`', and `sync --watch` uses fsnotify + OCI push rather than a git-commit loop — simpler, but a behavior change for --no-exit users.
- Compiled-in providers (kind/k3d/existing only) trade third-party provider velocity for zero plugin-protocol maintenance; Talos-in-docker or vcluster support means waiting for a release, not dropping in a binary.
- Data-only packs cannot express imperative logic; anything beyond Helm/Flux hooks (e.g., idpbuilder's dev-password API calls against Argo/Gitea) must be a Job manifest inside a pack or a Tier-2 exec plugin — uglier for the 10% case, in exchange for a near-zero attack surface and trivial authoring for the 90%.
- An in-cluster OCI registry is one more core moving part than 'nothing', and pushing artifacts is less familiar to users than `git push` to Gitea; we trade workflow familiarity for the deletion of an entire git-server subsystem and its trust/DNS pathologies.
- Committing to a v1 config schema at launch (vs k3d's years of v1alphaN) buys editor tooling and stability but raises the cost of early design mistakes — mitigated by shipping `neocube migrate` from day one.

**Risks:**
- Ecosystem adoption risk: CNOE stacks, docs, and muscle memory are Argo+Gitea shaped; if the cnoe-compat loader is less than excellent, neocube reads as 'a Flux tool' and loses the idpbuilder audience it targets.
- Helm v4 SDK is months old (GA Nov 2025); chart-v3 and SSA-default behaviors may churn under us — pin hard, wrap the action package behind one internal interface, and keep a helm-binary escape hatch out of the core.
- Modifying OS trust stores is security-sensitive and will trip corporate endpoint protection; `neocube trust` must be a separate, loudly-consented command with a clean uninstall, or it becomes our version of the cert-pain issue cluster.
- The 60-second promise depends on image pull times (Flux controllers + zot + kindest/node); without a pre-seeded node image or `neocube vendor` defaults, first-run experience on slow networks could replicate idpbuilder #229/#430 perception.
- Inventory-based pruning (fluxcd/pkg/ssa) deleting resources a user hand-edited or that a pack stopped declaring is the Crossplane 'unspecified resources get deleted' footgun — needs a prune-preview diff and an opt-out annotation from v0.1.
- Scope-creep pressure to add an RPC/Wasm plugin tier early (every platform tool gets this demand); the design only stays radically simple if maintainers hold the 'packs are data, providers are PRs' line until real evidence demands more.
- ksail already occupies the multi-provider/multi-engine niche; neocube must win on the deleted-complexity story (no CRDs, no git server, no daemon) and sub-minute UX, or it is a redundant entrant.

## Proposal: Neocube Facet Architecture — a provider-lattice CLI where every face of the platform (cluster, GitOps engine, git server, ingress, trust, packages) is a swappable provider behind a versioned interface

**Philosophy:** The local platform is not a demo — it is environment zero of the same platform you run in production, so the tool must behave like production tooling: declarative desired state, idempotent reconciliation, explicit versioning, and honest day-2 operations. Neocube therefore inverts idpbuilder's design at every point where idpbuilder chose 'compiled-in': there are no embedded components, only packages; no hardcoded trio, only a default profile; no create-once semantics, only reconcile-to-desired-state via server-side apply with an inventory; no laptop-resident controller pretending to be an operator, but a CLI that bootstraps a real GitOps engine and then gets out of the way. Everything a real platform team would need to swap — cluster substrate (kind, k3d, talos, an existing EKS context), GitOps engine (Flux or Argo CD), git server (Gitea, Forgejo, GitHub, GitLab, or none-at-all via OCI artifacts), ingress, and certificates — sits behind a small Go interface with at least two shipped implementations, because an abstraction with one implementation is a lie. Configuration is a typed, versioned CUE schema rather than a flag pile, packages are content-addressed OCI artifacts with a lockfile so installs are reproducible and air-gappable, and the CLI's own core stack is distributed through the exact same package mechanism users extend it with.

**Components:**
- **CLI shell (cmd/neocube)** (spf13/cobra v1.10 for the command tree; charm.land huh v2 for prompts (e.g. `neocube init` wizard); bubbletea v2 + bubbles v2 for the live bootstrap dashboard (per-component kstatus health, not a frozen spinner); log/slog with lipgloss v2 handler; fang/kong rejected to keep kubectl-style completion conventions.): Command tree, interactive prompts, progress UX, structured logging. Verb-first grammar (up/down/status/diff/upgrade/doctor/pkg/get) fixing idpbuilder's inconsistent grammar (#448), with a real progress model (#17) and quiet-by-default logs (#423/#438).
- **Config engine (pkg/config)** (cuelang.org/go v0.17 (pure Go, zero runtime deps) as the schema/validation core; sigs.k8s.io/yaml for YAML ingestion; cue.Value -> Go structs via cuelang encoding/gocode; JSON Schema exported with cue export --out jsonschema.): Loads, validates, defaults, and migrates the versioned Platform document. CUE is the native schema; users may write plain YAML which is unified against the CUE schema (typed errors, no `#{}`-style bolt-on templating). Publishes JSON Schema for editor autocomplete; explicit vN->vN+1 migration functions (the k3d lesson: never silently break v1alphaN).
- **Cluster providers (pkg/provider/cluster)** (sigs.k8s.io/kind/pkg/cluster v0.32 (default; docker/podman/nerdctl auto-detect); github.com/k3d-io/k3d/v5 typed config API; talos-in-docker via siderolabs Go packages; `existing` via k8s.io/client-go clientcmd (honors KUBECONFIG, never hardcodes ~/.kube/config — #362); vcluster as ephemeral-env-on-existing-cluster mode via Helm SDK driver. Out-of-tree providers pluggable via hashicorp/go-plugin (see extensibility).): One interface — Provision/Adopt/Kubeconfig/Teardown/Preflight — behind which local creation and existing/remote clusters are peers. `existing` provider adopts any kubeconfig context (kills idpbuilder #74/#246/#530); every provider implements Teardown with kubeconfig-context cleanup (#458) and Preflight (Docker socket, pid limits, inotify, disk — the #430 class of hangs becomes a pre-check failure with a message).
- **Apply & inventory engine (pkg/apply)** (fluxcd/pkg/ssa ResourceManager (SSA apply, kstatus health checks, wait, prune) — built exactly to replace kubectl; sigs.k8s.io/cli-utils inventory pattern; controller-runtime client for typed reads only. No custom CRDs required for core operation.): All cluster mutations go through one server-side-apply pipeline with health-checked waits, pruning, and a persistent inventory. The inventory (in-cluster Secret per platform) is what makes `up` idempotent, `down --cascade` complete (no orphaned registries/contexts — #351/#16), and `diff` capable of drift detection. Replaces idpbuilder's in-process controller-runtime manager + 3 CRDs: neocube is a CLI-driven reconciler; continuous reconciliation is the GitOps engine's job, not a laptop process.
- **GitOps engine providers (pkg/provider/gitops)** (Flux GitOps Toolkit semver Go APIs (source-controller, kustomize-controller, helm-controller types) + vendored flux2 v2.9 install manifests; Argo CD v3.4 driven via unstructured + its REST/gRPC pkg/apiclient only for session/password ops; health via kstatus in both cases.): Interface: Install, Bootstrap(repo|oci), CreateApp(pkg), Health, Uninstall. Flux is the default (library-grade, manifests vendorable at build time, works air-gapped); Argo CD is a first-class option but installed-not-embedded: neocube writes Application/AppProject CRs as unstructured against the live CRD schema — deliberately avoiding the stale argocd-api fork trap (#566) and Argo's unembeddable go.mod (#4055). A `none` engine exists for pure-SSA minimal setups.
- **Git server providers (pkg/provider/git)** (code.gitea.io/sdk/gitea (also serves Forgejo); google/go-github v66; gitlab.com/gitlab-org/api/client-go; go-git/go-git v5 for commit/push plumbing; oras-go v2 for the OCI path.): Interface: EnsureOrg/EnsureRepo/Push/CloneURLs{internal,external}/Credentials/Webhook. Ships gitea (in-cluster, default for laptop), forgejo, github, gitlab (answers #157/#186), and critically `none`: with Flux, local package content is pushed as OCI artifacts to the in-cluster registry and reconciled via OCIRepository — no git server needed at all. Admin user, org, branch are config, not literals (#304).
- **Package engine (pkg/pkg)** (helm.sh/helm/v4 SDK (action package, SSA installs) embedded in-process; sigs.k8s.io/kustomize/api; oras-go v2.6 for OCI pull/push with digest pinning; CUE for package manifests and values typing; kustomize-style remote git refs with commit pins (Dagger lesson) for compatibility with cnoe-io/stacks.): Everything installed — including the 'core' stack — is a package: an OCI artifact or git ref containing a manifest (neocube-pkg CUE/YAML), Helm chart refs + values, kustomizations, raw manifests, and health checks. Nothing is compiled into the binary (kills the #552/#565 'ingress EOL requires a fork' failure mode). A per-platform lockfile (neocube.lock: digests, image lists) gives reproducible installs, `upgrade --plan` diffs, and feeds air-gap bundling. Resolution order + priority is explicit in config, not flag-order magic (#548).
- **Trust & networking layer (pkg/net)** (crypto/x509 + smallstep/crypto for CA generation; FiloSottile/mkcert approach (truststore integration) vendored as a library; CoreDNS Corefile patch templates; Traefik v3 Helm chart as default ingress package; cert-manager package for ACME/BYO on remote.): Solves the two chronic idpbuilder wounds as one design: (a) single canonical hostname per service valid both inside and outside the cluster (generated CoreDNS rewrites + ingress), eliminating split-horizon URL breakage (#300/#398/#113); (b) a real CA story: generate a local CA once, install it into host trust store, containerd on nodes, and as a Secret for workloads (fixes #451/#545 OCI push trust), with byo-cert and cert-manager/ACME modes for remote clusters (#389). Ingress is a provider: traefik default (ingress-nginx is EOL 3/2026), nginx and gateway-api options.
- **Day-2 operations engine (pkg/ops)** (fluxcd/pkg/ssa dry-run diffing; kstatus for stuck-rollout root-causing (events + container statuses surfaced); Masterminds/semver for version constraint resolution; printers via lipgloss tables.): upgrade (resolve new package versions per constraints -> plan -> apply through the same SSA pipeline, with pre-upgrade CRD/version checks), diff/drift (SSA dry-run against inventory), doctor (preflights + live diagnosis: which deployment is stuck and why — the answer to #430's 57 comments), down --cascade (inventory-ordered deletion incl. external resources registered by packages, kube-context cleanup), and status (per-package health table).
- **Air-gap & registry component (pkg/airgap)** (oras-go v2 OCI image-layout; google/go-containerregistry for image copy/pull; zot (project.zotregistry.dev) as the in-cluster OCI registry package (replaces Gitea-as-registry double duty); kind's nodeutils image loading.): `neocube bundle create` walks the lockfile, pulls all charts/manifests/images into a single OCI layout tarball; `neocube up --bundle` installs fully offline, pre-loading images into kind/k3d nodes and serving artifacts from an in-cluster registry. First-class registry-mirror config passthrough. Directly answers #402/#501/#229/#520/#563/#564.
- **Plugin host (pkg/plugin)** (os/exec discovery of `neocube-*` on PATH (krew model, git-repo YAML index with sha256 pins); hashicorp/go-plugin v1.8 with VersionedPlugins + versioned protobuf for ClusterProvider/GitProvider/GitOpsEngine backends; extism/go-sdk v1.7 reserved for sandboxed validate/transform hooks, following Helm 4's HIP-0026 manifest format rather than inventing one.): Discovers and runs the three code-extension tiers: exec subcommands, gRPC provider backends, and (phase 2) Wasm hooks. Enforces digest pinning and prints krew-style trust warnings for exec plugins; negotiates protocol versions for gRPC providers.
- **Multi-cluster topology manager (pkg/fleet)** (client-go clientcmd for context fan-out; Flux remote-cluster kubeConfig reference pattern / Argo CD declarative cluster Secrets; same apply engine per cluster — no new machinery.): A Platform file may declare multiple clusters (e.g. mgmt + workload, or local + remote-staging) with per-cluster provider/engine/git settings and cross-cluster references (workload clusters register into the mgmt cluster's GitOps engine — Flux Kustomizations with remote kubeConfig secrets, or Argo cluster secrets). `neocube up` reconciles the whole topology; contexts are namespaced neocube-<platform>-<cluster>.

**Extensibility:** Four tiers, each with a concrete mechanism, ordered by how often users need them. (1) Declarative config is the core contract: a versioned `neocube.dev/v1alpha1 Platform` document validated by CUE, with published JSON Schema and automatic config-version migration — most 'extension' is just configuration (k3d/Devbox lesson). (2) Packages ('facets') are the primary unit: data-only bundles (CUE manifest + Helm chart refs/values + kustomizations + raw manifests + health checks + optional hooks) distributed as OCI artifacts via oras-go (Crossplane xpkg pattern, `neocube pkg add oci://ghcr.io/org/backstage:1.4.2`) or as git refs with resolved commit pins recorded in neocube.lock (Dagger lesson); the built-in 'core' stack (gitops engine, git server, ingress, dashboard) ships as a default profile of these same packages published to ghcr.io/neocube — dogfooding guarantees the mechanism stays honest. Packages need zero code, so authoring cost and attack surface are minimal. (3) Command-surface plugins are krew-style: any `neocube-<name>` executable on PATH becomes a subcommand; a git-repo index of YAML manifests with sha256-pinned per-platform archives, installed via `neocube plugin install`, with an explicit unaudited-plugin warning. (4) Provider backends — new cluster provisioners, git servers, GitOps engines — are hashicorp/go-plugin gRPC subprocesses speaking a versioned protobuf contract with VersionedPlugins handshake negotiation (Terraform lesson: keep the core protocol-agnostic behind pkg/provider interfaces; in-tree providers implement the same Go interfaces directly, out-of-tree ones arrive over gRPC). Tier 5 is tracked, not shipped: Extism Wasm hooks for sandboxed validators/transformers/post-renderers, adopting Helm 4's HIP-0026 plugin manifest format once it proves out. Deliberately avoided: CNAB-style bundle specs, a bespoke plugin registry (OCI + git suffice), an engine daemon, and unversioned subprocess contracts.

**Differentiators vs idpbuilder:**
- Existing, remote, and multi-cluster targets are first-class peers of kind — the `existing` cluster provider adopts any kubeconfig context, closing the ask idpbuilder deferred for 3 years (#74/#246/#530).
- Nothing is compiled in: ArgoCD/Gitea/ingress are not hardcoded trio-in-binary but ordinary versioned OCI packages in a profile — swapping ingress-nginx for Traefik is a one-line config change, not a fork (contrast #552/#565).
- Swappable GitOps engine (Flux default, Argo CD optional, none) and swappable git server (Gitea/Forgejo/GitHub/GitLab/none-via-OCI) behind Go interfaces, vs Argo-only + Gitea-only with a stale argocd-api fork (#566, #157).
- Real day-2: `neocube upgrade --plan`, drift `diff`, `doctor`, and inventory-driven `down --cascade` vs idpbuilder's destroy-and-recreate model with orphaned resources and stale contexts (#351/#458/#16/#489).
- Idempotent `up`: reconcile-to-desired-state via server-side apply + inventory, safe to re-run after config edits; no create-once immutability wall (`isCompatible` -> --recreate).
- One canonical URL per service inside and outside the cluster plus an installed-everywhere local CA (host trust store, containerd, workload Secret) — eliminates the split-horizon DNS and self-signed-cert issue clusters that dominate idpbuilder's tracker (#300/#398/#451/#545/#389).
- Reproducibility and air-gap by construction: digest lockfile + `neocube bundle` offline installs (#402/#501/#229 unaddressed in idpbuilder).
- Typed, versioned CUE-schema config with published JSON Schema and migrations, instead of 20 flags + nonstandard #{} templating.
- A designed extensibility model (data-only OCI packages, exec plugins, gRPC providers, Wasm hooks) instead of bolt-on customization limited to overlaying YAML on three hardcoded packages.
- No controller-runtime manager running on the laptop and no CRDs required for core operation — state lives in-cluster in the inventory; continuous reconciliation is delegated to the real GitOps engine, mirroring production operating models.
- Preflight + live diagnosis (`doctor`) turns the #430 'hangs forever with no clue' class into actionable errors with the failing pod, event, and fix hint.
- Compatibility bridge: a translator ingests cnoe:// ArgoCD Application packages and cnoe-io/stacks so existing content ports over.

**Tradeoffs:**
- Nothing embedded means first `up` needs network access to pull the default profile from ghcr.io — mitigated by digest lockfile caching and `neocube bundle`, but idpbuilder's fully-offline-binary property is traded for swappability and upgradeability.
- Flux-as-default trades Argo CD's beloved UI for library-grade embeddability and an air-gap-friendly bootstrap; Argo remains one config line away, but its integration is deliberately shallower (unstructured CRs, no compile-time coupling), so some deep Argo features (app-of-apps ownership takeover semantics) get less native support.
- CUE-native config raises the floor for advanced users' authoring experience but adds a learning curve; accepting plain YAML validated against the CUE schema softens this at the cost of maintaining two ergonomic paths.
- A provider matrix (4+ cluster providers x 2 engines x 5 git providers) multiplies the e2e test surface roughly 10x vs idpbuilder's single path — the price of the production-path promise, contained via blessed-profile CI rather than full-matrix CI.
- gRPC provider plugins (go-plugin) are per-OS binaries with zero sandboxing — accepted for the rare deep-integration case because Wasm can't yet host a kind-style provisioner that needs Docker sockets; the trust warning is explicit.
- CLI-driven reconciliation (no resident operator, no CRDs) keeps the laptop clean and mirrors 'flux bootstrap' semantics, but means drift between runs is only corrected when the user runs `up`/`diff` — continuous correction is delegated to the GitOps engine, which cannot self-heal the engine's own installation; an optional neocube-operator package is the future escape hatch.
- Refusing wire-compatibility with idpbuilder CRDs/cnoe:// (translator instead of native support) keeps the core clean but creates a migration step for existing CNOE users.
- Choosing Traefik as default ingress (nginx EOL-driven) breaks copy-paste familiarity with the huge ingress-nginx annotation corpus; nginx stays available as a package for those who need it.

**Risks:**
- Provider abstraction leakage: DNS, TLS trust, and port semantics differ enough between kind, k3d, talos-docker, and remote EKS that the ClusterProvider/net interfaces may sprout provider-specific escape hatches; mitigation is conformance tests per provider (a shared e2e suite every provider must pass) before accepting a fourth provider.
- Matrix explosion: 4 cluster providers x 2 GitOps engines x 5 git providers x 2 ingresses is untestable exhaustively; must define a small set of 'blessed profiles' tested in CI and mark the rest community-supported, or quality perception collapses on exactly the fragility complaints (#430-class) neocube exists to fix.
- Dependency youth: Helm v4 SDK is months past GA, bubbletea v2/charm.land paths are fresh, and Extism-based plugin patterns are unproven outside Helm — API churn here lands directly in neocube's core; pin aggressively and isolate behind internal wrappers.
- Driving Argo CD via unstructured CRs avoids the stale-fork trap but risks silent schema skew when Argo changes Application defaults (the #566 failure mode in a new costume); needs a version-probe that reads the live CRD schema and refuses unknown incompatible versions.
- Ecosystem gravity: CNOE stacks, docs, and mindshare are built around idpbuilder and cnoe:// — if the compatibility translator is less than excellent, neocube starts with zero installable content; the translator plus republishing popular stacks as neocube packages is launch-critical, not a nice-to-have.
- Trust-store modification (installing the local CA into host and containerd) is the most privileged thing the tool does and a security-sensitive surface; a bug here is a headline. Must be opt-in per store, fully reversible in `down`, and audited.
- Scope creep toward a platform product: multi-cluster + upgrade + drift + plugins is adjacent to Rancher/Kratix/vCluster-Platform territory; without a ruthless 'bootstrap and day-2 of the platform substrate, never workload orchestration' line, the project diffuses.
- Solo/early-team maintenance risk: idpbuilder shows ~2.5 years pre-1.0 with a focused scope; neocube's larger surface needs either corporate sponsorship or a deliberately staged roadmap (ship kind+existing, Flux, Gitea/none, one ingress first; add k3d/talos/Argo/GitLab after 1.0) to avoid the same stall.

## Proposal: Neocube "Lattice" — provider-matrix core, OCI-everything edges

**Philosophy:** Neocube treats the platform definition — not the tool — as the product: one typed, versioned, single-file blueprint (CUE) that a developer enjoys writing, which compiles deterministically onto any cluster (kind laptop, k3d, Talos, vcluster, or a real remote cluster) through a small library-grade core. Everything idpbuilder compiled into its binary becomes data at the edges: every component, including the 'core' GitOps engine, git server, gateway, and registry, is a digest-pinned, OCI-distributed package ('cube') that can be swapped, upgraded, or community-shared without forking. The core itself does only four things exceptionally well — provision, apply (server-side, idempotent, health-gated), wire trust/DNS so certs and split-horizon pain cannot exist, and narrate what it's doing through a rich TUI with typed, remediable errors. Reconciliation is delegated to a real GitOps engine (Flux by default, ArgoCD as an option) rather than a bespoke in-process operator, so `up` is idempotent, `upgrade` exists, and `down` is truly cascading. The DX bar is Dagger/Devbox/k3d: instant feedback (diff-before-apply, live health dashboard), a doctor that diagnoses instead of hanging, and an ecosystem where sharing a platform piece is `neocube push oci://...`.

**Components:**
- **CLI shell + TUI layer ("cockpit")** (spf13/cobra v1.10 for the command tree (kubectl-style completion + plugin discovery conventions); charm.land bubbletea v2 + bubbles v2 + lipgloss v2 for the live dashboard; huh v2 for `neocube init` wizard; stdlib slog with charmbracelet/log handler for plain mode): Command tree, interactive prompts, live progress rendering. Every long operation renders a Bubble Tea dashboard: per-component health (kstatus-derived), streaming events, elapsed time, and a hard deadline with an automatic failure digest — never an infinite 'waiting for deployment' spinner (idpbuilder #430/#17). Non-TTY/CI falls back to structured slog lines and --output json.
- **Config engine ("blueprint")** (cuelang.org/go v0.17 (pure Go, zero runtime deps); cue.Value unification for defaults/overlays; sigs.k8s.io/yaml for ingestion; invopop/jsonschema-style export generated from the CUE schema): Loads the single-file platform definition `neocube.cue` (apiVersion neocube.dev/v1alpha1, kind Platform), validates it against a typed CUE schema, exports a published JSON Schema for editor autocomplete, and performs explicit config-version migration (k3d lesson: migrate, then stabilize to v1 fast). YAML/JSON are accepted and internally unified into CUE values, so users never *have* to learn CUE — but get typed defaults, constraints, and cross-field validation when they do.
- **ClusterProvider layer** (sigs.k8s.io/kind/pkg/cluster (library Provider API, v0.32); github.com/k3d-io/k3d/v5 pkg/config/v1alpha5 + pkg/client; talos via siderolabs provision packages; existing-cluster via client-go clientcmd; out-of-tree providers via hashicorp/go-plugin (see extensibility)): Pluggable cluster lifecycle behind one Go interface: Create/Delete/Exists/Kubeconfig/LoadImages/PortMap. Built-in providers: kind (default), k3d, talos-docker, and — the #1 idpbuilder gap (#74) — `existing` (any kubeconfig context: EKS, GKE, on-prem; neocube skips provisioning and only installs/wires the platform). vcluster provider gives ephemeral platforms on a shared cluster. Container runtime handling is delegated to kind/k3d's own docker/podman/nerdctl detection — no bespoke docker client coupling.
- **Apply/reconcile engine ("loom")** (fluxcd/pkg/ssa ResourceManager (SSA + WaitForSet + kstatus health + prune) on controller-runtime client; sigs.k8s.io/cli-utils kstatus; NO in-process CRD controllers — plain library calls, deterministic exit codes, per-step timeouts): neocube's own bootstrap applies: server-side apply with ownership, health waiting via kstatus, pruning, and drift-tolerant re-runs. `neocube up` is idempotent — running it twice converges instead of erroring or drifting passwords (idpbuilder #489/#296). Also powers `neocube diff` (server-side dry-run diff before any change) for instant feedback.
- **GitOpsEngine abstraction** (fluxcd/flux2 embedded install manifests + GitOps Toolkit API packages (source-controller, kustomize-controller, helm-controller types); argoproj/argo-cd consumed only as rendered manifests + pkg/apiclient over gRPC, never as a go.mod library dependency): Ongoing reconciliation is handed to a real GitOps engine, selected in config: Flux (default, because the GitOps Toolkit is library-grade and its manifests embed cleanly) or Argo CD (installed-not-embedded, driven via its manifests + gRPC apiclient, kept for its UI). Packages compile to Flux Kustomization/HelmRelease or Argo Application objects from the same neutral package model — no cnoe://-style engine-coupled URI scheme leaking into user YAML.
- **GitProvider layer** (code.gitea.io/sdk + go-git/go-git v5 for the embedded default; google/go-github and gitlab.com/gitlab-org/api/client-go for external providers; Flux OCIRepository for the git-less mode via oras-go pushes): Where the platform's desired state lives. Default: in-cluster Gitea (batteries included, like idpbuilder). But it's an interface — GitHub, GitLab, and generic-remote providers ship in-tree (idpbuilder #157), and `provider: none` mode lets Flux source straight from OCI artifacts with no git server at all (great for CI). Repo org/user, branch, and commit identity are all configurable (kills the giteaAdmin/main hardcoding, #304).
- **Trust & networking fabric** (smallstep/crypto (minica-style CA) + adapted mkcert truststore logic (filippo.io/mkcert internals, Go); Envoy Gateway via Gateway API (sigs.k8s.io/gateway-api types), Traefik as alternate package; zot (project-zot/zot) as the in-cluster OCI registry package instead of overloading Gitea): Solves idpbuilder's two chronic warts structurally. (1) TLS: a per-user local CA created once, installed into the OS trust store (opt-in, mkcert-style) AND injected into the cluster's containerd trust — so `docker push` to the platform registry and in-cluster pulls both just work (#451/#545); BYO cert is a config field (#389). (2) Split-horizon DNS: one canonical hostname scheme (default *.neocube.localtest.me, path-routing on by default per #187) used identically inside and outside the cluster via generated CoreDNS rewrites, so webhooks and registry pushes see the same URLs (#300/#398). Ingress is Gateway API-first with Envoy Gateway default (ingress-nginx is EOL, #552) behind a swappable `ingress` package.
- **Package system ("cubes") + rendering** (oras-go v2 for OCI push/pull (Crossplane xpkg pattern: plain OCI image, annotated layers); helm.sh/helm/v4 pkg/action for in-process chart rendering/install (SSA by default); sigs.k8s.io/kustomize/api; CUE for cube parameter schemas and validation; Masterminds/semver for dependency resolution): A cube is a data-only, OCI-distributed package: cube.cue metadata (name, semver, params schema as CUE, dependencies), plus content = Helm chart ref + values, kustomize overlay, or raw manifests, plus optional service/ingress/secret declarations and doctor checks. Rendering happens in-process (Helm v4 SDK, kustomize api) — no helm/kubectl binaries needed at runtime, but also no compile-time-embedded stack: even 'core' pieces (gitops engine, gitea, gateway, registry) are just cubes pinned by digest in a default catalog, so swapping or upgrading any of them is config, not a fork (#5/#126/#565).
- **Doctor & diagnostics** (Custom check framework (each cube can ship checks as data: HTTP probe, TLS probe, resource-health selector); kstatus + client-go events for diagnosis; error taxonomy rendered with lipgloss; goroutine-parallel checks with hard timeouts): `neocube doctor` runs preflight (container runtime reachable, podman vs docker quirks, pid/inotify/ulimit limits — #433, free ports, disk, DNS resolution of the chosen host, registry reachability/rate limits — #229) and postflight (per-cube health, cert chain validation from both host and in-cluster viewpoints, git/registry round-trip test). Every failure maps to a typed error code (NEO1234) with a one-line cause, a remediation command, and a docs URL. `neocube doctor --bundle` produces a redacted support tarball (events, logs, config, versions).
- **State & lifecycle manager** (fluxcd/pkg/ssa inventory-style object tracking; client-go for the in-cluster inventory record; adrg/xdg + BoltDB (go.etcd.io/bbolt) for the local index; clientcmd for kubeconfig context add/remove hygiene): Tracks what neocube created where, enabling the verbs idpbuilder lacks: `upgrade` (re-resolve cube digests, diff, apply with health gates), `down` (cascading delete: cluster AND registry containers AND kubeconfig contexts AND locally-installed CA if requested — #351/#458/#16), and `status`. State lives in-cluster (a single Secret/ConfigMap inventory per platform, SSA field ownership as source of truth) plus a tiny local index at ~/.neocube for cluster discovery — no fragile local statefile as authority.
- **Offline/air-gap bundler** (oras-go v2 + OCI image-layout archives; google/go-containerregistry (crane libraries) for image copy/resolve; kind/k3d image-load APIs): `neocube bundle create` resolves the full platform (cubes, charts, container images via chart image-list extraction) into one OCI archive; `neocube up --bundle platform.tar` provisions with zero internet, pre-loading images into the cluster via the provider's LoadImages. Registry mirrors and pull-through cache are first-class config (#402/#501/#563/#229).

**Extensibility:** Four tiers, cheapest first, each with a concrete mechanism. Tier 1 — versioned declarative config: the neocube.cue Platform schema (apiVersion neocube.dev/v1alpha1) with published JSON Schema and automatic version migration; most 'extension' is just configuration. Tier 2 (primary) — cubes: data-only packages distributed as OCI artifacts via oras-go (Crossplane xpkg layout: plain OCI image, cube.cue metadata layer + content layers holding Helm chart refs/values, kustomize overlays, raw manifests, ingress/secret/doctor-check declarations). Consumed as `oci://ref@semver-or-digest` or git refs with commit pins (Dagger lesson); resolved with Masterminds/semver; parameters are CUE-typed so `neocube add` can interactively prompt (huh) and validate before touching the cluster. Community catalog is just any OCI registry plus an index repo — no bespoke registry service. This covers ~80% of extensions with zero code and near-zero attack surface. Tier 3 — exec plugins for new command surface: any `neocube-<name>` binary on PATH becomes `neocube <name>` (kubectl/krew model), with a krew-style git-repo index of sha256-pinned per-platform archives and an explicit unaudited-plugin warning; plugins receive context via env (NEOCUBE_PLATFORM_CONFIG, NEOCUBE_KUBECONFIG, NEOCUBE_OUTPUT_FORMAT). Tier 4 — gRPC provider plugins for deep integration only (custom ClusterProviders, GitProviders, secret backends): hashicorp/go-plugin subprocesses with VersionedPlugins handshake negotiation and a versioned protobuf contract (neocube.provider.v1), keeping the core protocol-agnostic like Terraform. Future tier tracked, not bet on: Extism Wasm hooks for sandboxed validators/transformers/post-renderers, adopted only once Helm 4's HIP-0026 system proves out — the cube format reserves a `hooks` layer media type for it so no format break is needed.

**UX:**
Commands: `neocube init` (huh wizard → writes neocube.cue), `neocube up [-f neocube.cue]` (Bubble Tea dashboard: per-cube spinners → health checkmarks, hard timeouts, failure digest with NEO-code + fix), `neocube diff` (server-side dry-run diff of blueprint vs cluster), `neocube status` (live TUI or --output json), `neocube doctor [--bundle]`, `neocube add oci://ghcr.io/neocube/cubes/backstage@2 --set org=acme`, `neocube upgrade [--cube gitea]`, `neocube bundle create -o platform.tar` / `neocube up --bundle platform.tar`, `neocube secrets get argocd`, `neocube down --cascade`, `neocube push oci://ghcr.io/acme/cubes/mycube`. Config (neocube.cue):\n\n  apiVersion: \"neocube.dev/v1alpha1\"\n  kind:       \"Platform\"\n  metadata: name: \"acme-dev\"\n  cluster: {\n    provider: \"kind\"            // kind | k3d | talos | vcluster | existing\n    version:  \"1.33\"\n    // provider: \"existing\", context: \"arn:aws:eks:...\" for remote\n  }\n  network: {\n    host: \"acme.localtest.me\"   // one canonical name, in- and out-of-cluster\n    pathRouting: true            // default\n    tls: localCA: trustOS: true  // or tls: certFile/keyFile for BYO\n  }\n  gitops: engine: \"flux\"         // flux (default) | argocd\n  git:    provider: \"gitea\"      // gitea (in-cluster) | github | gitlab | none (OCI source)\n  cubes: {\n    backstage: {\n      source: \"oci://ghcr.io/neocube/cubes/backstage@2.1.0\"\n      params: { org: \"acme\", auth: \"guest\" }\n    }\n    myapp: {\n      source: \"./deploy\"         // local dir, live-synced to git provider\n      params: replicas: 2\n    }\n  }\n\nCUE gives autocomplete (exported JSON Schema), typed params validated before any cluster call, and comments/defaults; plain YAML with the same shape is accepted for CUE-averse users.

**Differentiators vs idpbuilder:**
- Works on ANY cluster, not just kind: ClusterProvider matrix (kind/k3d/talos/vcluster/existing-kubeconfig) directly answers idpbuilder #74, its oldest and most-demanded gap — the same neocube.cue provisions a laptop kind cluster today and installs onto a shared EKS cluster tomorrow.
- No compiled-in stack: idpbuilder's argocd/gitea/nginx trio is go:embed'd and required a source fork to swap the EOL ingress (#565); in neocube every component — including the GitOps engine, git server, gateway, and registry — is a digest-pinned OCI cube in a default catalog, swappable and upgradeable by editing one file.
- Real lifecycle verbs: idempotent `up`, first-class `upgrade`, `diff` before apply, and cascading `down` that cleans kube contexts, containers, and trust material — idpbuilder's model is create-once/delete-and-recreate with orphaned residue (#351, #458, #489).
- Trust and DNS solved structurally, not documented around: a locally-trusted CA installed in the OS and containerd truststores plus a single canonical hostname inside and outside the cluster eliminates the largest open issue cluster in idpbuilder (#451, #545, #389, #300, #398).
- DX as a feature: Bubble Tea live progress with per-component health and hard timeouts instead of a silent infinite wait (#430/#17), a `doctor` command with typed error codes and remediation, and huh-powered `neocube init` — idpbuilder has none of these.
- Flux-library core instead of ArgoCD coupling: fluxcd/pkg/ssa and GitOps Toolkit are semver'd embeddable Go modules, avoiding idpbuilder's stale forked argocd-api CRDs (#566) and the cnoe:// scheme; ArgoCD remains available as a cube for its UI, installed not linked.
- Ecosystem by design: cubes are shareable OCI artifacts with CUE-typed parameters and semver dependencies (`neocube add oci://ghcr.io/acme/cubes/backstage@1.2.0`), versus idpbuilder's -p flag pointing at raw ArgoCD Application YAML in git paths with priority-index conflict rules (#548).
- Air-gapped and CI-friendly from day one: bundle export/import, registry mirrors, and a git-less OCI-source mode — idpbuilder pulls everything from the internet on every create (#402, #229).
- Typed single-file config users enjoy: CUE schema with editor autocomplete via published JSON Schema, defaults, and cross-field validation, replacing a 17-flag CLI invocation plus a nonstandard #{} templating dialect (#336).

**Tradeoffs:**
- CUE-native config vs familiarity: CUE is markedly better than YAML+#{} templating for typed, composable platform definitions, but has a learning curve; mitigated by accepting plain YAML, shipping `neocube init`, and exporting JSON Schema — yet two accepted syntaxes mean two documentation paths.
- Flux as default engine vs ArgoCD's UI gravity: Flux is the only library-grade choice (ArgoCD's go.mod is unembeddable, issue #4055), but many platform engineers expect the ArgoCD UI; we pay for supporting both engines behind one abstraction with a permanent compatibility-matrix cost.
- Nothing compiled in vs offline-by-default: idpbuilder's embedded manifests work with zero network for core install; neocube resolves cubes from OCI at run time, so first-run requires network or a pre-built bundle — we trade the go:embed guarantee for swappability and buy back offline via the bundler.
- Provider matrix vs depth: five cluster providers multiply the test surface (podman quirks × k3d × talos × existing-cluster RBAC variance); we accept tiered support (kind/existing = tier-1 CI-gated, others = tier-2) rather than pretending uniform quality.
- OS truststore modification is powerful but invasive: trusting the local CA requires sudo/keychain access; it is strictly opt-in (trustOS: true) with a documented no-trust fallback (exported CA path), keeping the magic without silent system mutation.
- Library core, not in-cluster operator: doing bootstrap as plain SSA library calls (no neocube CRDs/controllers) gives deterministic exits and simple debugging, but forgoes continuous in-cluster self-healing of neocube's own wiring — we delegate that to Flux/ArgoCD and accept that neocube-the-CLI must be re-run to change platform shape.
- Four extensibility tiers add conceptual surface vs krew-only simplicity: we accept the documentation burden because data-only cubes (tier 2) intentionally absorb most demand, keeping code plugins rare.

**Risks:**
- CUE adoption risk: if the community rejects CUE authoring despite the YAML on-ramp, the differentiating config layer becomes a liability; watch conversion rates in telemetry-free ways (docs traffic, issue language) and keep the YAML path first-class.
- Dependency churn: Helm v4 SDK (GA Nov 2025) and Charm v2 (new charm.land paths) are young; Timoni-style CUE+OCI patterns are pre-1.0 — pin aggressively, wrap Helm/Charm behind thin internal interfaces, and budget for breaking-change releases in year one.
- Ecosystem cold start: cubes are only valuable with a catalog; without ~20 high-quality cubes at launch (backstage, argocd, gitea, envoy-gateway, zot, crossplane, kargo, external-secrets...) the OCI story is theoretical — seed and CI-test an official catalog before v0.1.
- Competitive squeeze: ksail already covers the provider/engine matrix and idpbuilder has CNOE/CNCF gravity; neocube must win on DX polish and the cube ecosystem, not feature checklists, or it becomes a third also-ran.
- Existing-cluster mode expands the blast radius: installing gateways, CAs, and GitOps engines onto shared/remote clusters can collide with tenant workloads and RBAC; requires a conservative default profile (namespaced install, no cluster-scoped mutations without --allow-cluster-scope) or early adopters will get burned.
- Trust-fabric security surface: shipping a tool that mints CAs and edits OS truststores invites scrutiny; a mistake here is reputationally fatal — needs an external security review and a threat model doc before 1.0.
- Scope creep toward production platform management: the same pull that idpbuilder resisted (#74 debates) will hit neocube from the other side; guard the identity as 'bootstrap and day-1/day-2 of dev-grade platforms', delegating day-N fleet management to Flux/ArgoCD/Kratix rather than growing an operator empire.
- Multi-engine abstraction leakage: keeping one cube model that compiles to both Flux and ArgoCD objects risks lowest-common-denominator semantics (sync waves vs dependsOn); mitigate with engine-specific escape hatches in cube.cue and conformance tests per engine.
