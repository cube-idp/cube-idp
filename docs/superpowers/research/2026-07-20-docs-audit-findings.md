## Provenance

This report is the output of the phase-1 oracle audit. How it was produced:

- **241 agents** ran across the audit: an inventory sweep over eight comment batches plus a docs batch, seven parallel audit clusters, and two independent adversarial verifiers per raw finding.
- **7 clusters** partitioned the corpus by surface: `cli-readme`, `cli-docs`, `diag-registry`, `packs`, `plugins`, `web`, and the comment-site inventory sweep.
- Every finding was checked against `hack/truth-index.json` (the oracle: commands, diagCodes, configSchema, packContract, exitContract, outputSchemas) and against the code itself — never against another document.
- **2 adversarial verifiers per finding**, each instructed to refute. A finding dies only on a 2/2 refutation; a 1/2 refutation keeps it, flagged `⚠ contested`.
- **117 raw findings → 109 survived → 8 refuted** (2/2). Of the 109 survivors, **28 are contested** (1/2 refuted) and carry a footnote with the dissent.

### How to read this

Findings are bucketed by *what kind of defect they are*, and the bucket determines what may be done about them. **`suspected-bug`** means the documentation is right and the code is wrong — these are report-only, no doc edit is proposed, and they need operator triage as separate work. **`stale-doc`** means the code moved and the prose did not; the doc is edited to match code. **`dangling`** means a reference (file path, anchor, command, code) that no longer resolves. **`planning-leak`** means process residue — phase numbers, gate IDs (`GT15`), work-package IDs (`WP8`, `P7`), task numbers, `decision N` markers, spec-section pointers — that leaked from planning documents into shipped prose or shipped source comments; these are stripped and replaced with the product-meaningful statement. **`unverifiable`** means the claim is about something outside the repository's reach and neither the oracle nor the code can adjudicate it; the operator decides. Note carefully: **`CUBE-####` is the product's diagnostic-code namespace, not planning residue** — the 108 diagnostic codes are a shipped, user-facing API surfaced by `cube-idp explain`, and no finding proposes touching a code identifier. What findings *do* touch is planning tokens that got embedded *inside* those codes' human-readable summaries.

Every row carries a `Decision:` suffix for the operator to annotate in place at GATE 1.

## 1. `suspected-bug` — 2 findings (REPORT ONLY, operator triage required)

**No documentation edit is proposed for these.** In both cases the doc states the intended contract and the *manifest* deviates from it. Fixing them means changing code, which is outside the scope of this audit's phases 2–5. They are recorded here in full so the operator can route them to separate triage.

Both findings are the same defect in two packs: an HTTPRoute hardcodes the development gateway hostname `cube-idp.localtest.me` instead of using the contract-designated `${GATEWAY_FQDN}` substitution token, while the pack's `expose.urls` advertises `${GATEWAY_HOST}`. On any cube whose `spec.gateway.host` is not the development default, the advertised URL has no matching route.

### 1.1 `packs · packs/argocd/pack.cue:9`

- **Claim (doc):** `expose.urls: ["https://argocd.${GATEWAY_HOST}"]` — the pack advertises its UI at the cube's configured gateway host.
- **Code reality:** The pack's only HTTPRoute hardcodes `hostnames: ["argocd.cube-idp.localtest.me"]` instead of `argocd.${GATEWAY_FQDN}`. Every other route-carrying pack in the repo (argo-workflows, backstage, floci, floci-ui, kargo, prometheus-stack) uses `${GATEWAY_FQDN}`. On any cube whose `spec.gateway.host` is not `cube-idp.localtest.me`, the advertised expose URL does not route.
- **Code ref:** `packs/argocd/manifests/20-httproute.yaml:16`
- **Proposed resolution:** report only — no doc edit; route to code triage.
- **Confidence:** high

**Verifier 1 — not refuted:**

> Verified against code and contract; the finding holds. (1) Doc claim is real: /Users/rafal.pieniazek/github.com/cube-idp/packs/packs/argocd/pack.cue line 10 (finding cites 9, where the `expose: {` block opens — off-by-one, cosmetic) reads `urls: ["https://argocd.${GATEWAY_HOST}"]`. (2) codeRef is real and codeReality is accurate: /Users/rafal.pieniazek/github.com/cube-idp/packs/packs/argocd/manifests/20-httproute.yaml:16 reads `hostnames: ["argocd.cube-idp.localtest.me"]` — a hardcoded literal with no token. (3) The mismatch is load-bearing: CONTRACT.md:106-107 defines `${GATEWAY_HOST}` as host[:port] and `${GATEWAY_FQDN}` as the bare host "for Gateway API hostnames: fields", and CONTRACT.md's Substitution section states manifests/*.yaml are substituted as plain-text replacement on raw bytes. Plain-text replacement cannot rewrite a hardcoded literal, so on any cube whose spec.gateway.host != cube-idp.localtest.me the advertised expose URL has no matching HTTPRoute hostname and does not route. (4) The file is demonstrably token-aware — line 15 uses `${GATEWAY_PACK}` in parentRefs — so line 16's literal is an omission, not a rendering-mode artifact. (5) Attempted refutation via the peer-pack comparison only partially lands: the claim "every other route-carrying pack uses ${GATEWAY_FQDN}" is slightly overstated because packs/gitea/manifests/20-httproute.yaml:21 also hardcodes `gitea.cube-idp.localtest.me` while packs/gitea/pack.cue:10 advertises `${GATEWAY_HOST}`. That is a second instance of the same defect, widening the bug rather than refuting it; argo-workflows:23, backstage:18, floci:22, floci-ui:23, kargo:27, prometheus-stack:25 all do use `${GATEWAY_FQDN}` as stated. Bucket suspected-bug is correct: the pack.cue doc statement expresses the intended contract and the manifest is the deviation, so the fix belongs in code, not the doc.

**Verifier 2 — not refuted:**

> Finding survives adversarial verification. (1) Not torn from context: packs/argocd/pack.cue:9-10 is a live `expose` block, not a roadmap/changelog/illustrative section; the adjacent D11 comment concerns the auth secret, not the URL. (2) The divergence is real: packs/argocd/manifests/20-httproute.yaml:16 hardcodes `hostnames: ["argocd.cube-idp.localtest.me"]` while pack.cue advertises `https://argocd.${GATEWAY_HOST}`. (3) `${GATEWAY_FQDN}` is a genuine, contract-documented substitution — CONTRACT.md:106-107 designates `${GATEWAY_FQDN}` as the bare host specifically "for Gateway API `hostnames:` fields", and internal/pack/expose.go:63 (`strings.ReplaceAll(s, "${GATEWAY_FQDN}", gw.Host)`) plus internal/pack/render.go:29 apply it to manifests. (4) The majority pattern confirms intent: argo-workflows, backstage, floci, floci-ui, kargo and prometheus-stack all use `${GATEWAY_FQDN}` in their HTTPRoute hostnames, each with an explicit comment justifying the FQDN-vs-HOST split (e.g. packs/kargo/manifests/10-httproute.yaml:4-6). argocd's HTTPRoute comment justifies only the SSA/OutOfSync explicit-defaults choice and never claims the literal hostname is deliberate — the missing standard justification is evidence of oversight, not of deliberate spec. (5) No CUBE-#### planning code is involved. Bucket `suspected-bug` is correct: the spec (pack.cue expose + CONTRACT.md) is authoritative and the manifest fails to honor it, so the code is wrong rather than the doc being stale. Additional scope worth passing upstream: packs/gitea/manifests/20-httproute.yaml:21 has the identical defect (`gitea.cube-idp.localtest.me` vs packs/gitea/pack.cue:10's `https://gitea.${GATEWAY_HOST}`), and packs/argocd/README.md:17 propagates the hardcoded hostname into prose. This broadens the finding to two packs rather than refuting it.

**Decision:** _(operator: approve for separate triage / reject / defer)_

### 1.2 `packs · packs/gitea/pack.cue:10`

- **Claim (doc):** `expose.urls: ["https://gitea.${GATEWAY_HOST}"]` — the pack advertises Gitea at the cube's configured gateway host.
- **Code reality:** packs/gitea/manifests/20-httproute.yaml hardcodes `hostnames: ["gitea.cube-idp.localtest.me"]` rather than `gitea.${GATEWAY_FQDN}`, so on a cube configured with any other `spec.gateway.host` the expose URL (and `cube-idp get secrets`' advertised endpoint) points at a hostname no route matches. The same file is cited elsewhere in the repo as the reference HTTPRoute pattern, which propagates the divergence.
- **Code ref:** `packs/gitea/manifests/20-httproute.yaml:21`
- **Proposed resolution:** report only — no doc edit; route to code triage.
- **Confidence:** high

**Verifier 1 — not refuted:**

> Finding survives adversarial verification on every checkable axis. (1) Claim location verified: packs/gitea/pack.cue:10 is literally `urls: ["https://gitea.${GATEWAY_HOST}"]`. (2) codeRef verified: packs/gitea/manifests/20-httproute.yaml:21 is literally `hostnames: ["gitea.cube-idp.localtest.me"]` — hardcoded, no token. (3) codeReality is accurate: CONTRACT.md:107 defines `${GATEWAY_FQDN}` as "the bare host (no port) — for Gateway API `hostnames:` fields", i.e. the contract-designated token for exactly this field, which gitea does not use. Tellingly, line 20 of the SAME file DOES use `namespace: ${GATEWAY_PACK}`, so the omission is inconsistent within the file itself, not an unaware-of-substitution artifact. (4) No rescue mechanism: CONTRACT.md §3 defines substitution as plain-text replacement on raw manifest bytes; a literal containing no `${...}` token is never rewritten, at any gateway host. (5) Not house style — an isolated divergence: backstage, kargo, argo-workflows, floci, floci-ui, prometheus-stack all correctly use `X.${GATEWAY_FQDN}`; only gitea and argocd hardcode `cube-idp.localtest.me`. (6) Propagation claim verified: six files cite packs/gitea/manifests/20-httproute.yaml as the reference HTTPRoute pattern (argocd/manifests/20-httproute.yaml:2, floci/manifests/30-httproute.yaml:4, prometheus-stack/manifests/10-httproute.yaml:4, argo-workflows/manifests/30-httproute.yaml:4, backstage/manifests/20-httproute.yaml:3, envoy-gateway/manifests/20-gateway.yaml:7) — and argocd, the closest citer, inherited the same hardcoding. (7) Oracle confirms `spec.gateway.host` is a user-configurable string field (configSchema, tag `host`), so the mismatch is reachable, not hypothetical. Additional supporting detail the auditor did not mention: the defect is masked in CI because both conformance fixtures (hack/conformance_config.tmpl.yaml:7 and hack/conformance_config_gateway.tmpl.yaml:7) pin `host: cube-idp.localtest.me`, so the hardcoded literal coincidentally matches and no test fails. Bucket `suspected-bug` is correct — the doc/contract is right and the manifest is wrong, so this is a code defect rather than a stale doc.

**Verifier 2 — not refuted:**

> Finding survives adversarial verification; bucket confirmed as suspected-bug. DOC-INTENT CHECK (fair quotation): packs/gitea/pack.cue:10 is a live `expose` block, not roadmap/changelog/illustrative example. CONTRACT.md:95 defines expose.urls as the operative D11 discoverability record ("may carry ${GATEWAY_*} tokens"), and CONTRACT.md §3 states substitution applies to `expose.urls` "when URLs are displayed or recorded". Quoted fairly. REFUTATION ATTEMPT 1 — is the literal hostname correct via some other mechanism? No. CONTRACT.md:107 is explicit that ${GATEWAY_FQDN} exists precisely "for Gateway API hostnames: fields, which cannot carry ports". There is no reverse mapping from a literal hostname to spec.gateway.host. hack/conformance_config.tmpl.yaml:7 shows cube-idp.localtest.me is only the TEST default (spec.gateway.host), not a guaranteed value. REFUTATION ATTEMPT 2 — deliberate spec rather than bug? This is the strongest evidence FOR the finding. Only gitea (line 21) and argocd (packs/argocd/manifests/20-httproute.yaml:16) hardcode the literal. A deliberate choice would have been annotated — that file annotates everything else in a 9-line header comment, which justifies only the explicit-schema-defaulted-fields rule and is silent on the hostname. REFUTATION ATTEMPT 3 — CUBE-#### planning residue? Not applicable, no diagnostic code involved. CORRECTION to the finding's reasoning (does not change verdict): the finding claims the file "is cited elsewhere as the reference HTTPRoute pattern, which propagates the divergence." The citations exist but reference it narrowly for the explicit-schema-defaulted-fields rule, and several citing files explicitly contrast their own CORRECT usage — e.g. packs/floci/manifests/30-httproute.yaml:4-6 reads "see packs/gitea/manifests/20-httproute.yaml). hostnames via D15's ${GATEWAY_FQDN}". So the divergence has NOT propagated. The core defect is unaffected. BUCKET: suspected-bug is correct, not stale-doc. pack.cue's expose.urls is the correct/contract-conformant side; the manifest is the wrong side. Fix belongs in code (packs/gitea/manifests/20-httproute.yaml:21 -> "gitea.${GATEWAY_FQDN}", and same for packs/argocd/manifests/20-httproute.yaml:16), not in the doc. Note packs/gitea/README.md:4 and packs/argocd/README.md:17 also repeat the literal hostname and would need updating alongside the code fix.
