## idpbuilder-internals

**Summary:** idpbuilder (github.com/cnoe-io/idpbuilder, CNOE project, latest release v0.10.2, 2026-04-13) is a single Go binary that stands up a complete local Internal Developer Platform with only Docker as a dependency: it creates a kind cluster, then runs an in-process controller-runtime manager with three controllers (Localbuild, GitRepository, CustomPackage) that install embedded manifests for ingress-nginx, ArgoCD and Gitea, push those manifests into in-cluster Gitea repos, and hand ongoing management to ArgoCD via GitOps. Custom packages are ArgoCD Application/ApplicationSet YAMLs passed via `-p/--package` (dir, file, or remote URL), where the `cnoe://` repoURL scheme makes idpbuilder push local directories into Gitea and rewrite the Application's repoURL to the in-cluster repository.

**Key findings:**
- End-to-end flow of `idpbuilder create`: parse cobra flags -> create kind cluster from templated pkg/kind/resources/kind.yaml.tmpl (default node image kindest/node:v1.33.1, host port 8443->443, fixed 32222 for git SSH) -> apply 3 embedded CRDs -> set up CoreDNS override and self-signed TLS cert (SAN *.cnoe.localtest.me) -> start controller-runtime manager in-process -> create a cluster-scoped Localbuild CR (named after --name, default 'localdev') and let the reconcilers do the rest.
- The binary itself is the operator: pkg/controllers/run.go registers LocalbuildReconciler, gitrepository.RepositoryReconciler and custompackage.Reconciler with a controller-runtime manager running on the developer's machine (metrics bound to '0'), not in the cluster; when everything is synced the Localbuild controller calls CancelFunc and the CLI exits (unless --no-exit).
- Core packages are hard-coded to exactly three: v1alpha1.ArgoCDPackageName='argocd', GiteaPackageName='gitea', IngressNginxPackageName='nginx'. They are installed in parallel goroutines (installCorePackages) from manifests embedded with go:embed at pkg/controllers/localbuild/resources/{argo/install.yaml, gitea/k8s/install.yaml, nginx/k8s/ingress-nginx.yaml}; those files are generated at build time by hack/argo-cd/generate-manifests.sh (kustomize), hack/gitea/generate-manifests.sh (helm template with hack/gitea/values.yaml -> Deployment 'my-gitea' in namespace 'gitea'), and hack/ingress-nginx/.
- After core install, LocalbuildReconciler.ReconcileArgoAppsWithGitea 'hands control to ArgoCD': for each of argocd/nginx/gitea it creates a GitRepository CR with Source.Type=embedded, the GitRepository controller pushes the embedded manifests into a Gitea repo (org 'giteaAdmin', repo named <namespace>-<name>), and an ArgoCD Application in namespace 'argocd' (project 'default') is created pointing at status.internalGitRepositoryUrl — so users can thereafter change platform components via git push to Gitea.
- Three CRDs under group idpbuilder.cnoe.io/v1alpha1 (YAML in pkg/controllers/resources/): localbuilds (cluster-scoped; spec.packageConfigs + spec.buildCustomization; status tracks ArgoCD/Gitea/Nginx availability and Gitea URLs/admin secret), gitrepositories (spec.source.type local|embedded|remote, provider gitea|github, status.internal/externalGitRepositoryUrl + latest commit hash), custompackages (spec.argoCD.applicationFile/name/namespace/type Application|ApplicationSet, gitServerURL, remoteRepository, replicate).
- -p/--package accepts local directories, single YAML files, and remote URLs (kustomize-style 'https://github.com/org/repo//path?ref=...'); helpers.ParsePackageStrings splits them; each YAML must be an Argo CD Application or ApplicationSet (isSupportedArgoCDTypes checks group argoproj.io); the Localbuild controller creates one CustomPackage CR per app, annotated with cnoe.io/package-priority = its -p index (processed in reverse so later -p flags win conflicts for the same app name, including deleting lower-priority CustomPackages and taking over GitRepository ownership).
- The cnoe:// URI scheme (v1alpha1.CNOEURIScheme) is the local-dev mechanism: in an Application's spec.source(s).repoURL, ApplicationSet git generators (including matrix generators) and even helm valuesObject strings, 'cnoe://<relative-path>' is resolved relative to the application file, a GitRepository CR of type 'local' is created, the directory content is committed/pushed to Gitea by author 'git-reconciler <idpbuilder-agent@cnoe.io>' on branch 'main', and the repoURL is rewritten to e.g. http://my-gitea-http.gitea.svc.cluster.local:3000/giteaAdmin/<repo>.git.
- Exit semantics: shouldShutDown() only returns true when all Applications labeled cnoe.io/package-type=core are Healthy AND every GitRepository/CustomPackage with a cnoe.io/cli-start-time annotation matching this CLI invocation has Status.Synced; the GitRepository controller requeues every 30s and re-commits on local file changes, so `--no-exit` (-n) gives continuous local-dir-to-Gitea syncing.
- Full `create` flag surface (pkg/cmd/create/root.go): --recreate, --name/--build-name (deprecated) default 'localdev', --dev-password (sets static password 'developer' for argocd+gitea admins via Gitea SDK AdminEditUser and ArgoCD /api/v1/session + /account/password), --kube-version (v1.33.1), --extra-ports, --kind-config, --registry-config, --registry-mirrors (target=address), --insecure-registry-mirrors, --host (default cnoe.localtest.me), --ingress-host-name, --protocol (https), --port (8443), --use-path-routing, -p/--package, -c/--package-custom-file (<pkg>:<file>, only 'argocd', 'gitea', 'nginx' accepted), -n/--no-exit.
- Core-package customization is deliberately narrow: -c/--package-custom-file overlays user YAML onto the embedded manifests via k8s.BuildCustomizedObjects (EmbeddedInstallation.installResources); everything else about argocd/gitea/nginx (namespaces 'argocd'/'gitea'/'ingress-nginx', versions, Gitea admin user 'giteaAdmin', secret handling, Argo project 'default') is fixed in the binary; BuildCustomizationSpec fields are immutable after cluster creation (Build.isCompatible errors and tells you to --recreate).
- Other subcommands: `idpbuilder get clusters|packages|secrets` (secrets uses a different -p meaning package name; secrets must be labeled cnoe.io/cli-secret=true and cnoe.io/package-name=<name>), `idpbuilder delete` and `idpbuilder version`; pkg/printer renders the output tables.
- A GitHub provider (v1alpha1.GitProviderGitHub, pkg/controllers/gitrepository/github.go) exists alongside Gitea in the GitRepository controller, but Localbuild/CustomPackage flows always set Provider.Name=gitea — GitHub is not reachable through the CLI's normal path.
- Design history is documented in docs/pluggable-packages.md: ArgoCD is a declared hard requirement, Gitea replaced an earlier embedded Go git server, and manifests are pre-rendered at build time (helm template/kustomize in hack/) precisely so the runtime needs no helm/kustomize binaries.
- Codebase is small and focused (~190 files): main.go, api/v1alpha1 (types), globals (ProjectName 'idpbuilder', DefaultHostName 'cnoe.localtest.me'), pkg/cmd (cobra), pkg/build (orchestration+coredns+tls), pkg/kind (cluster lifecycle via kind-as-library), pkg/controllers/{localbuild,gitrepository,custompackage}, pkg/k8s (YAML->object decoding with Go-template rendering), pkg/util (gitea client, go-git clone/push, RepoMap mutexes), pkg/printer, tests/e2e.

## 1. What idpbuilder does end-to-end

[idpbuilder](https://github.com/cnoe-io/idpbuilder) (CNOE / CNCF sandbox ecosystem, latest release **v0.10.2**, 2026-04-13) is a single Go binary whose promise is "spin up a complete internal developer platform with only Docker required as a dependency." `idpbuilder create` does, in order (`pkg/build/build.go` `Build.Run`):

1. **kind cluster** — `pkg/kind/cluster.go` uses kind as a library with a Go-templated config (`pkg/kind/resources/kind.yaml.tmpl`): one control-plane node (`kindest/node:{{.KubernetesVersion}}`, default `v1.33.1`), host port `8443`→container `443` (or 80 for `--protocol http`), a fixed `32222` mapping for Gitea SSH, optional `--extra-ports`, containerd `certs.d` mounts for `--registry-mirrors`/`--registry-config`. `--kind-config` replaces the template entirely.
2. **CRDs** — `controllers.EnsureCRDs` applies three embedded CRD YAMLs from `pkg/controllers/resources/`: `localbuilds.idpbuilder.cnoe.io`, `gitrepositories.idpbuilder.cnoe.io`, `custompackages.idpbuilder.cnoe.io`.
3. **CoreDNS + TLS** — patches CoreDNS (templates in `pkg/build/templates/coredns/`) so in-cluster names resolve for the chosen host, and creates a self-signed cert (secret/CM `idpbuilder-cert`, SAN `*.cnoe.localtest.me`).
4. **Controllers in-process** — starts a controller-runtime `Manager` on the laptop (not in-cluster) and creates a cluster-scoped `Localbuild` CR named after `--name` (default `localdev`), annotated with `cnoe.io/cli-start-time`.

The **LocalbuildReconciler** then bootstraps: creates namespace `idpbuilder-<name>`, installs the three core packages **in parallel** (`installCorePackages`), waits for readiness, then "passes control to argocd": for each of `argocd`, `gitea`, `nginx` it creates a `GitRepository` CR (`Source.Type=embedded`), whose controller pushes the very manifests used for installation into a Gitea repo, and creates an ArgoCD `Application` pointing at the in-cluster repo URL. From then on the platform manages itself via GitOps; users can edit core components by pushing to Gitea. Default URLs: `https://argocd.cnoe.localtest.me:8443` and `https://gitea.cnoe.localtest.me:8443` (or `/argocd`, `/gitea` with `--use-path-routing`); `localtest.me` resolves to 127.0.0.1. Credentials: `idpbuilder get secrets -p argocd`.

## 2. Codebase structure and the localbuild controller

Key packages (~190 files): `api/v1alpha1` (CRD Go types), `globals` (constants: `ProjectName=idpbuilder`, `ArgoCDNamespace=argocd`, `NginxNamespace=ingress-nginx`, `DefaultHostName=cnoe.localtest.me`), `pkg/cmd` (cobra: `create`, `delete`, `get {clusters,packages,secrets}`, `version`), `pkg/build`, `pkg/kind`, `pkg/controllers/{localbuild,gitrepository,custompackage}`, `pkg/k8s` (decode YAML→typed/unstructured objects, render Go templates against `BuildCustomizationSpec`), `pkg/util`, `pkg/printer`, `tests/e2e`.

**Manifest embedding:** manifests are rendered at *build* time by `hack/` scripts — ArgoCD via kustomize (`hack/argo-cd/`), Gitea via `helm template` (`hack/gitea/values.yaml`; produces Deployment `my-gitea` in namespace `gitea`), ingress-nginx similarly — into `pkg/controllers/localbuild/resources/`, then compiled in with `//go:embed resources/argo/*` etc. (`argo.go`, `gitea.go`, `nginx.go`). The `EmbeddedInstallation` struct (`installer.go`) applies them, optionally overlaying a user customization file, and polls `monitoredResources` (e.g. Deployment `argocd-server`, StatefulSet `argocd-application-controller`, Deployment `my-gitea`) with a 5-minute timeout.

**Reconciliation:** `LocalbuildReconciler.Reconcile` (971-line `controller.go`) is a classic controller-runtime reconciler (`For(&v1alpha1.Localbuild{})`) with sub-reconcilers, 15s requeue. It also implements `--dev-password` (rewrites ArgoCD admin password via REST `/api/v1/session` + `/account/password`, and Gitea via the Gitea SDK, to static `developer`). Exit logic: `shouldShutDown` requires all `cnoe.io/package-type=core` Applications Healthy and all `GitRepository`/`CustomPackage` resources from *this* CLI invocation (matched via `cnoe.io/cli-start-time`) synced; it then annotates Argo apps with `argocd.argoproj.io/refresh: normal` and cancels the context so the CLI exits (unless `--no-exit`).

**GitRepository controller** (`pkg/controllers/gitrepository/`): pluggable `gitProvider` (gitea via `code.gitea.io/sdk`, plus an unused-by-CLI github provider). It creates the repo (`<namespace>-<name>` under org `giteaAdmin`), materializes content from embedded FS / local path / remote repo path, commits as `git-reconciler <idpbuilder-agent@cnoe.io>` on branch `main` using go-git, pushes, and records `status.internalGitRepositoryUrl`. Requeues every 30s, committing only when the worktree is dirty — this is what gives live local-directory sync.

## 3. Custom packages / `-p`

`idpbuilder create -p <dir|file|remote-url>` (repeatable). Each YAML must be an ArgoCD `Application` or `ApplicationSet`. Localbuild creates a `CustomPackage` CR per app with annotations `cnoe.io/package-priority` (the `-p` index; lists are processed in reverse so **later `-p` wins** for the same app name — lower-priority CustomPackages are deleted and GitRepository ownership is taken over) and `cnoe.io/package-source-path`. The **CustomPackage controller** parses the app file; any `repoURL` (in `spec.source(s)`, ApplicationSet git/matrix generators, or nested strings inside `helm.valuesObject`) beginning with **`cnoe://`** is treated as a path relative to the app file: a local-type `GitRepository` is created, content is pushed to Gitea, and the URL is rewritten to e.g. `http://my-gitea-http.gitea.svc.cluster.local:3000/giteaAdmin/<repo>.git`. Non-`cnoe://` URLs pass through untouched. Remote `-p` URLs use kustomize remote syntax (`https://github.com/org/repo//path?ref=v1.2.3`) — this is how CNOE "stacks" (github.com/cnoe-io/stacks: backstage, terraform integrations, etc.) are consumed.

## 4. Core abstractions

- **Localbuild** (cluster-scoped): the whole desired platform — `spec.packageConfigs` (argo enabled, embedded apps enabled, customPackage{Files,Dirs,Urls}, `packageCustomization` map) and `spec.buildCustomization` (protocol/host/ingressHost/port/usePathRouting/selfSignedCert/staticPassword/registryMirrors — compared by `isCompatible`, immutable per cluster).
- **GitRepository**: desired in-Gitea repo; `spec.source.type` `embedded|local|remote`, `spec.provider` (gitea|github, org `giteaAdmin`), status has internal/external clone URLs + latest commit.
- **CustomPackage**: one user Argo app; `spec.argoCD{applicationFile,name,namespace,type}`, git server URLs, `remoteRepository`, `replicate`; status.synced + gitRepositoryRefs.

## 5. Hard-coded vs configurable

**Hard-coded:** the core trio and their manifests/versions (change requires editing `hack/` and recompiling), namespaces (`argocd`, `gitea`, `ingress-nginx`), Gitea admin `giteaAdmin` + secret `gitea-credential`, branch `main`, commit author, SSH port 32222, Argo project `default`, static password `developer`, CRD group `idpbuilder.cnoe.io`, project namespace pattern `idpbuilder-<name>`. **Configurable:** everything on the `create` flag list (`--name`, `--recreate`, `--kube-version`, `--extra-ports`, `--kind-config`, `--registry-config`, `--registry-mirrors`, `--insecure-registry-mirrors`, `--host`, `--ingress-host-name`, `--protocol`, `--port`, `--use-path-routing`, `--dev-password`, `-p/--package`, `-c/--package-custom-file` — the latter restricted to `argocd|gitea|nginx`, format `name:/path/to/file.yaml` — and `-n/--no-exit`).

## pain-points

**Summary:** idpbuilder (cnoe-io/idpbuilder, 323 stars, v0.10.2, still pre-1.0 after ~2.5 years) is deliberately scoped as a local, kind-only demo/dev tool with a compiled-in ArgoCD + Gitea + ingress-nginx stack, and its issue tracker shows the same friction themes recurring for years: fragile kind/Docker bootstrap (#430, 57 comments, open), self-signed-certificate and OCI-registry trust pain (#451, #545, #534, #389), no support for existing/remote clusters (#74, open since 2023), no pluggable git provider or GitOps engine (#157, #32), weak teardown/idempotency (#351, #458, #489, #296), and no real upgrade or air-gapped story (#566, #402, #229). Maintainers have repeatedly deferred the "real cluster / production path" direction, which is the strongest argument for replacing rather than extending it.

**Key findings:**
- Kind-only by design: docs (cnoe.io/docs/idpbuilder) scope it to a local kind cluster with Docker as the only dependency; issue #74 'Support for external clusters?' has been open since 2023 with maintainers explicitly deferring it past a 'tentative v1.0.0', and #246 'Extend idpbuilder create to work with existing k8s cluster' was closed as a duplicate of #74 rather than implemented. #530 'CNOE on Cloud instance' remains open.
- The single most-commented issue is #430 'Waiting for Deployment ingress-nginx-controller or my-gitea to become ready never ends' (57 comments, open since Nov 2024) — bootstrap hangs forever with no timeout or diagnostics when kind/Docker/host environment is off (also #433 pid limits, #107, #434 'Need an easy way to debug issues').
- Hardcoded embedded stack: core packages (ArgoCD, Gitea, ingress-nginx) are compiled into the Go binary (pkg/apps embedded resources); design tension exists since issue #5 'Design pluggable distro packaging mechanism' (2023), and #126 'Provide a way to customize core packages' is still open. #552/#565 show ingress-nginx EOL (March 2026) forcing a Traefik replacement PR because the ingress controller was not swappable.
- Gitea-only git backend: #157 'add integration with GH, GH enterprise, GitLab' open with zero progress; #186 (GitHub templates), #304 (hardcoded giteaAdmin username), #296 (deleting gitea pod resets password). ArgoCD is the only supported GitOps engine — no Flux option, and the custom cnoe:// URL scheme (#355) plus stale cnoe-io/argocd-api CRD fork (#566 'Old ArgoCD dependencies cause issues') deepen the coupling.
- Certificate/TLS pain is the biggest recurring open cluster: self-signed CA breaks docker push to the Gitea OCI registry (#451), needs manual containerd trust (#545), no BYO certificate (#389), cert export workarounds (#555, #328, #500), Backstage-Gitea trust issues (#127, #316), corporate-proxy TLS interception breaks image pulls (#520).
- In-cluster vs external DNS mismatch is a chronic architectural wart: #300 'CoreDNS configuration for core packages' (16 comments), #113, #398 'Support gitea as internal registry' (16 comments) — URLs differ inside and outside the cluster, breaking webhooks and image pushes.
- No air-gapped/offline story: #402 'Use local images if available', #501 (import images from host, still draft), #563/#564 (registry mirrors), #487 (registry-config), #229 (Docker Hub rate limits block install) — every create pulls ArgoCD/redis/nginx/gitea images from the internet.
- Weak teardown/idempotency: #351 'Cascading delete' (cloud resources orphaned on delete), #458 (kube contexts left in kubeconfig after delete), #16 (docker registry not deleted), #489 'Improve reconcilers for detecting password drift', #440 (create doesn't exit when applying new packages).
- No upgrade story: component bumps are manual maintainer chores (#422 'update gitea and argocd', #498 ArgoCD 2.10→3.x, #384 K8s version, #382 'Automate documenting component version updates'); there is no 'idpbuilder upgrade' — the model is destroy-and-recreate.
- CLI/UX friction backlog: hardcoded ~/.kube/config (#362 open), inconsistent command grammar (#448), noisy logs (#423, #438, #512, #387), no progress indication (#17), path-routing not default despite ref-implementation needing it (#187), --package override-order confusion (#548).
- Podman/non-Docker support is a long-running 20-comment effort (#473, #472, #85) that still isn't first-class; Codespaces/devpod needed dedicated fixes (#148, #386, #452, #269).
- Customization is bolt-on: templating was added late with a nonstandard #{ } delimiter to avoid Helm/Go-template collisions (#336, #295); custom CNI (#453) and cloud-credential passthrough (#245) remain open.
- The maintainers themselves debated renaming the project to 'localidp'/'idplocal' (#52, 15 comments, still open) because 'idpbuilder... does not necessarily support building a production grade idp' — an explicit acknowledgment the tool's identity is local-demo, not platform bootstrap.
- Ecosystem signal: 94 open issues, 323 stars, v0.10.2 released 2026-04-13, no ADOPTERS page yet (#504); CNOE's own blog (cnoe.io/blog/intro-to-idpbuilder) positions it for demos, PoCs, CI testing and local development, with production 'needing further adjustments'.

## Scope and positioning

idpbuilder (https://github.com/cnoe-io/idpbuilder) is the CNOE project's CLI that spins up a local IDP: a kind cluster plus three core packages — ArgoCD, Gitea (with embedded OCI registry), and ingress-nginx — wired together with a self-signed wildcard cert, CoreDNS rewrites, and the `cnoe.localtest.me` domain (per https://cnoe.io/docs/idpbuilder). As of 2026-07 it sits at v0.10.2 (2026-04-13), 323 stars, 94 open issues. CNOE's own blog (https://cnoe.io/blog/intro-to-idpbuilder) frames it for demos, PoCs, CI integration testing, and local platform-engineer iteration — explicitly not production. Issue #52 ("rename the repo for idpbuilder to idplocal or localidpbuilder", open, 15 comments) captures the maintainers themselves saying the name over-promises: it builds a *local* IDP, not a production one.

## Recurring complaints and friction

- **Bootstrap fragility.** The most-commented issue in the repo, #430 "Waiting for Deployment ingress-nginx-controller or my-gitea to become ready never ends" (57 comments, open since Nov 2024), shows `idpbuilder create` looping forever with no timeout, no failure diagnosis, and maintainers hand-debugging users' Docker environments in comments. Related: #433 (pid limits kill nginx), #107, #434 "Need an easy way to debug issues", #71 (unfriendly error when Docker isn't running).
- **Certificate/registry trust.** A dense cluster of open issues: #451 (docker push to Gitea registry fails cert verification), #545 (containerd inside cluster doesn't trust the CA), #534 (OCI image SDLC with self-signed certs), #389 (no BYO certificate), #555/#328/#500 (cert export/import/document workarounds), #127/#316 (Backstage↔Gitea trust), #353 (cert validity too long).
- **Split-horizon DNS.** In-cluster vs host URLs differ, breaking webhooks and pushes: #300 "CoreDNS configuration for core packages" (16 comments), #398 "Support gitea as internal registry" (16 comments), #113.
- **Slow feedback loop.** #297: default ArgoCD 3-minute refresh is too slow for local dev; annotation-based refresh misses generated apps.
- **CLI/UX debt.** #362 (kubeconfig path hardcoded to ~/.kube/config), #448 (inconsistent command grammar), #423/#438/#512/#387 (log noise), #17 (no progress bar), #187 (path-routing should be default), #548 (--package override order), #445/#449 (cluster listing).

## Architectural limitations

- **Kind/Docker-only.** #74 "Support for external clusters?" (open since 2023) is the canonical thread; #246 was closed into it; #530 asks for cloud-instance deploys. Maintainer comments in #74 show deliberate deferral ("not the next release, but the one after, tentative v1.0.0") — v1.0 still hasn't shipped. Podman support is a grind (#473 with 20 comments, #472, #85); Codespaces needed special-casing (#148, #386, #452).
- **Embedded, hardcoded stack.** Core packages are compiled into the binary; #5 (2023) already flagged the need for a pluggable "distro" mechanism; #126 (customize core packages) is still open. The ingress-nginx EOL (March 2026) forced #552/#565 — replacing the controller wholesale via PR because it wasn't swappable; #453 (custom CNI) open.
- **ArgoCD-only, Gitea-only.** No Flux path; the `cnoe://` scheme (#355) and the stale forked cnoe-io/argocd-api CRDs (#566, open — out-of-sync syncPolicy defaults corrupt user apps) hard-couple it to a specific ArgoCD vintage. Git provider is Gitea, full stop: #157 (GitHub/GHE/GitLab integration) open with no progress, #304 (hardcoded giteaAdmin), #186.
- **Upgrade story: none.** Version bumps are manual maintainer PRs (#422, #498 ArgoCD 2.10→3.x, #384 K8s bump, #382); no upgrade command; the operating model is delete-and-recreate, which collides with the idempotency gaps below.
- **Air-gapped/offline: unsupported.** #402 (use local images), #501 (import from host — still an experiment), #563/#564 (registry mirrors), #487 (registry-config), #229 (Docker Hub rate limits break install), #520 (corporate TLS interception breaks pulls).
- **Teardown/idempotency.** #351 (delete orphans cloud resources created by the platform), #458 (stale kube contexts after delete), #16 (registry container survives delete), #489 (password drift breaks reconcilers on re-run), #296 (gitea pod restart loses password), #440 (create with new packages never exits).
- **Multi-cluster:** effectively absent; only #5's closed design thread and GitOps-Bridge musings inside #74 touch it.

## Resurfacing feature requests

Existing-cluster support (#74/#246/#530), external git providers (#157/#186), BYO certificates (#389), registry mirrors/local images (#402/#501/#563/#564/#487), custom package lifecycle commands (#469/#466), secrets surfacing (#193/#464/#491), dev-mode static passwords (#441/#442 → regressed in #519), templating (#295/#336 — shipped with a nonstandard `#{}` delimiter), Homebrew/Chocolatey packaging (#231, #506 brew shipped stale versions).

## Why replace rather than extend

1. **The maintainers guard the local-only scope.** #74 and #52 show consensus that idpbuilder is a local ideation tool; "production path" requests get parked. Extending it to real clusters fights the project's stated identity.
2. **Extensibility is bolted on, not designed in.** Core packages, cert handling, CoreDNS rewrites, kind config, and the Gitea/ArgoCD wiring are embedded in Go code; #5's pluggability design was never generalized, so every swap (ingress in #565, CNI in #453, git provider in #157) is a code fork, not configuration.
3. **Coupling debt.** The private argocd-api fork (#566), `cnoe://` scheme, giteaAdmin literals (#304), and localbuild controller assumptions mean an extension inherits stale dependencies and split-horizon hacks.
4. **Lifecycle gaps are structural.** No upgrade, no cascading delete, password drift on re-run — a tool intended for durable platforms needs idempotent reconciliation from day one, which idpbuilder's create-once model lacks.
5. **Still pre-1.0 after ~2.5 years** with 94 open issues and modest velocity on the architectural asks — the roadmap pressure is toward polishing the local demo, not toward the existing-cluster/multi-provider space.

## ecosystem-scan

**Summary:** The 2026 Go/cloud-native ecosystem is well-suited to a next-gen local IDP-builder: kind (v0.32.0) and k3d (v5.9.0) are cleanly embeddable Go libraries; Flux's GitOps Toolkit packages plus fluxcd/pkg/ssa give a library-grade GitOps/apply engine (unlike Argo CD, which is hostile to embedding); Helm 4 (helm.sh/helm/v4, GA Nov 2025) ships a stable SDK, defaults to server-side apply, and pioneered Extism-Wasm plugins distributed over OCI — a pattern a new tool should copy. CUE (v0.17.0, pure Go) is the strongest embeddable config language; cobra + Bubble Tea v2 + huh v2 is the de-facto CLI/TUX stack; and idpbuilder itself (v0.10.2, Apache-2.0) remains active but architecturally conservative, with ksail emerging as the closest multi-distro successor.

**Key findings:**
- Cluster provisioning: kind v0.32.0 (Apache-2.0, 2026-06-02) exposes sigs.k8s.io/kind/pkg/cluster as a supported Go library (Provider API for create/delete/kubeconfig) and supports docker/podman/nerdctl via KIND_EXPERIMENTAL_PROVIDER with auto-detection — the safest embed target.
- k3d v5.9.0 (MIT, 2026-06-02) is importable as github.com/k3d-io/k3d/v5 with a typed config API (pkg/config/v1alpha5); k3s itself is at v1.36.2+k3s1 (Apache-2.0, CNCF) but is a distro, not a library — you embed it via k3d.
- Talos v1.13.6 (MPL-2.0) offers 'talosctl cluster create docker' for production-parity local clusters; vcluster v0.35.1 (Apache-2.0 core, but freemium: embedded etcd/private nodes gated behind Platform license) needs a host cluster and its Go SDK targets plugin syncers, not programmatic cluster creation.
- GitOps: Flux is the embeddable option — GitOps Toolkit controllers ship semver Go API packages, flux2 v2.9.1 embeds install manifests at build time, and fluxcd/pkg/ssa provides a standalone server-side-apply ResourceManager (built to replace kubectl entirely); flux-operator v0.54.1 (ControlPlane) adds FluxInstance CRD + web UI.
- Argo CD v3.4.5 is effectively not embeddable as a library: its go.mod requires k8s.io/kubernetes directly with replace directives (issue #4055), so the viable integrations are installing its manifests + gRPC apiclient — which is exactly what idpbuilder does.
- Helm 4 (GA at KubeCon NA Nov 2025, now v4.2.3) established helm.sh/helm/v4 as a stable, restructured Go SDK (pkg/action), defaults to server-side apply on install, and rebuilt plugins as Extism/Wasm sandboxed modules installable from OCI registries (helm plugin install oci://...) with signature verification.
- Config languages: CUE v0.17.0 (Apache-2.0, pure-Go cuelang.org/go) embeds with zero non-Go dependencies; KCL is CNCF Sandbox with core at v0.11.2 (last stable Apr 2025, Rust core bound via kcl-lang/lib native/wasm bindings); Pkl 0.32.0's pkl-go spawns the pkl CLI as a child process — a heavy runtime dependency for a Go CLI.
- Timoni v0.27.0 (Apache-2.0, 2026-07-02, CUE v0.12 engine, CRD import incl. oneOf/anyOf) is the best existing demonstration of CUE-modules-as-OCI-artifacts for Kubernetes, but is explicitly pre-1.0 with unstable APIs.
- CLI frameworks: cobra v1.10.2 (Apache-2.0) remains the Kubernetes-ecosystem default; urfave/cli v3.10.1 (MIT) is leaner and now stable; the Charm stack went v2-GA — bubbletea v2.0.8, lipgloss v2, bubbles v2, huh v2.0.3 under new charm.land module paths, with the new 'Cursed' renderer and Mode-2026 synchronized output.
- Plugins: hashicorp/go-plugin v1.8.0 (MPL-2.0, RPC subprocess, battle-tested by Terraform/Vault) vs Extism v1.30.0 (BSD-3-Clause, sandboxed Wasm, go-sdk v1.7.1) — Helm 4's adoption of Extism + OCI distribution (via oras-go v2.6.2) is the strongest signal for the Wasm+OCI route; krew v0.5.0 shows exec-style kubectl plugins remain viable as low-friction fallback.
- Container runtimes: no unified Go client abstraction exists — the pragmatic 2026 pattern (testcontainers-go, kind) is Docker-API-socket detection via DOCKER_HOST/podman socket plus CLI-provider fallback covering docker, podman v6.0.1 (daemonless), and nerdctl v2.3.4 (containerd).
- idpbuilder (CNOE, Apache-2.0) latest release is v0.10.2 (2026-04-13; ArgoCD v3.1.7, kind v0.29, registry mirrors) with repo activity into July 2026 — architecture is kind + Gitea (with OCI registry) + Argo CD + ingress-nginx with manifests baked into the binary at compile time.
- Alternatives/successors: ksail (devantler-tech/ksail, Go, pkg.go.dev-published) is the closest next-gen contender — one CLI across Kind/K3d/Talos/VCluster/KWOK/EKS with Flux or ArgoCD, SOPS/age secrets, and an MCP server; Kratix (Syntasso, Apache-2.0) covers in-cluster platform-API orchestration (Promises) rather than local bootstrap; Score is now CNCF Sandbox (donated July 2024) as a workload-spec ingestion format.
- Recommended composite stack for a new tool: kind pkg/cluster (+ optional k3d/talos providers behind an interface), Flux GitOps Toolkit + fluxcd/pkg/ssa for reconciliation with Argo CD as an optional installed-not-embedded UX layer, CUE (cuelang.org/go) + Helm v4 SDK for packaging, cobra + Bubble Tea/huh v2 for CLI/TUX, and Extism-Wasm plugins distributed as OCI artifacts via oras-go with exec-plugin fallback.

## 1. Local cluster provisioning

- **kind v0.32.0** (Apache-2.0, released 2026-06-02) — mature CNCF SIG project explicitly designed as "Go packages implementing cluster creation... along with a CLI built on these packages." [`sigs.k8s.io/kind/pkg/cluster`](https://pkg.go.dev/sigs.k8s.io/kind/pkg/cluster) gives a `Provider` for create/delete/kubeconfig; [cloud-provider-kind](https://github.com/kubernetes-sigs/cloud-provider-kind) adds LoadBalancer support. Supports docker/podman/nerdctl node providers (auto-detected; `KIND_EXPERIMENTAL_PROVIDER` to force). **Adopt as default provider.**
- **k3d v5.9.0** (MIT, 2026-06-02) — importable as `github.com/k3d-io/k3d/v5` with typed config (`pkg/config/v1alpha5`); wraps **k3s v1.36.2+k3s1** (Apache-2.0, CNCF). Good second provider for lightweight/multi-node.
- **Talos v1.13.6** (MPL-2.0, 2026-07-09) — `talosctl cluster create docker` runs full Talos userspace (apid/machined/kubelet) in containers; best production-parity story, config patches supported; embed via Sidero's Go packages or exec. MPL-2.0 is fine for linking.
- **vcluster v0.35.1** (Apache-2.0 core, 2026-06-19) — needs a host cluster (so it's a tenancy layer, not a bootstrap layer); creation goes through the Helm driver; the Go SDK ([loft-sh/vcluster-sdk](https://github.com/loft-sh/vcluster-sdk)) is for plugin syncers. Note the [OSS-vs-Free split](https://www.vcluster.com/docs/vcluster/introduction/oss-vs-free): embedded etcd, private nodes etc. require Platform license validation.

**Adopt:** a `ClusterProvider` interface with kind (library) as default, k3d (library) and talos-in-docker as alternates, vcluster as an "ephemeral env on existing cluster" mode — exactly the matrix [ksail](https://github.com/devantler-tech/ksail) proved viable.

## 2. GitOps engines

- **Flux v2.9.1** (Apache-2.0; Flux 2.8 GA Feb 2026) — the [GitOps Toolkit APIs](https://fluxcd.io/flux/gitops-toolkit/packages/) are semver'd Go modules; flux2 CLI embeds controller manifests at build time (installs offline). [`fluxcd/pkg/ssa`](https://pkg.go.dev/github.com/fluxcd/pkg/ssa) is a standalone SSA `ResourceManager` Flux built specifically to eliminate kubectl. [flux-operator v0.54.1](https://github.com/controlplaneio-fluxcd/flux-operator) adds FluxInstance lifecycle + web UI.
- **Argo CD v3.4.5** — hostile to library embedding ([go.mod requires k8s.io/kubernetes with replaces, issue #4055](https://github.com/argoproj/argo-cd/issues/4055)); drive it by installing manifests and using `pkg/apiclient` gRPC, as idpbuilder does. Keep it optional for its UI.
- **Plain SSA apply** — `k8s.io/client-go/applyconfigurations` + controller-runtime `client.Apply`; `fluxcd/pkg/ssa` is the highest-level ready-made wrapper (waits, health checks via kstatus, pruning).

**Adopt:** fluxcd/pkg/ssa for the tool's own bootstrap applies; Flux toolkit controllers as the default reconciler; Argo CD as an installable option, never a compile-time dependency.

## 3. Config languages

- **CUE v0.17.0** (2026-06-29, Apache-2.0) — pure Go (`cuelang.org/go`), zero runtime deps, big recent typechecker perf gains (up to 80% on large projects), OCI module registry. **Best embed.**
- **Timoni v0.27.0** (2026-07-02, Apache-2.0) — CUE-powered Helm alternative with OCI modules and CRD schema import; still pre-1.0/unstable API — steal patterns, depend cautiously.
- **KCL** (CNCF Sandbox since Sept 2023; core v0.11.2, last stable Apr 2025) — Rust core; `kcl-lang.io/kcl-go` binds via native lib/wasm ([kcl-lang/lib](https://github.com/kcl-lang/lib)) — heavier embed, slower core cadence.
- **Pkl 0.32.0** (Apple, Apache-2.0) — [pkl-go spawns the pkl CLI as a child process](https://pkl-lang.org/go/current/evaluation.html); poor fit for a single-binary Go CLI.
- **Helm 4 SDK** (`helm.sh/helm/v4`, GA KubeCon NA Nov 2025, now v4.2.3) — restructured stable SDK, slog logging, SSA-by-default installs, chart v3. Needed regardless for ecosystem charts.
- **YAML+kustomize** — keep as ingestion format via `sigs.k8s.io/kustomize/api`; table stakes, not differentiation.

**Adopt:** CUE as native config layer; Helm v4 SDK + kustomize for compatibility; watch KCL.

## 4. CLI frameworks

**cobra v1.10.2** (Dec 2025, Apache-2.0) remains the k8s-ecosystem standard; **urfave/cli v3.10.1** (MIT, June 2026) is stable and leaner but loses ecosystem conventions (kubectl-style completion, plugin discovery). Charm stack is now **v2 GA**: bubbletea v2.0.8, lipgloss v2, bubbles v2, huh v2.0.3 under `charm.land/*` paths, new "Cursed" renderer + synchronized output ([v2 notes](https://github.com/charmbracelet/bubbletea/discussions/1374)). **Adopt:** cobra for command tree; huh v2 for interactive prompts; bubbletea v2 for progress/dashboards — all MIT/Apache.

## 5. Plugin systems

- **hashicorp/go-plugin v1.8.0** (MPL-2.0) — RPC subprocess model, proven by Terraform/Vault; heavyweight, plugins are per-OS binaries.
- **Extism v1.30.0** (BSD-3-Clause; go-sdk v1.7.1) — sandboxed Wasm, deny-by-default host access. **Helm 4 adopted exactly this (HIP-0026): Extism/v1 runtime plugins distributed via OCI (`helm plugin install oci://...`) with provenance verification** ([helm.sh plugins overview](https://helm.sh/docs/plugins/overview/)).
- **OCI distribution** — `oras-go v2.6.2` (2026-07-10) is the standard library; Flux/Timoni/CUE/KCL all publish artifacts to OCI already.
- **Exec/krew-style** (krew v0.5.0) — zero-infrastructure fallback, no sandboxing.

**Adopt:** Extism Wasm plugins distributed as OCI artifacts (oras-go), plus `mytool-*` exec-plugin discovery for escape hatches. Follow Helm 4's manifest format rather than inventing one.

## 6. Container runtime abstraction

No unified Go abstraction exists. Proven pattern (kind, [testcontainers-go](https://golang.testcontainers.org/system_requirements/using_podman/)): talk Docker Engine API over auto-discovered socket (`DOCKER_HOST`, podman's docker-compatible socket) and/or shell out to a detected provider CLI. Podman v6.0.1 (daemonless) and nerdctl v2.3.4 (containerd, needs >=1.7 for kind) both work with kind today. **Adopt:** delegate runtime handling to kind/k3d providers; document podman/nerdctl as supported-experimental; avoid hard docker-client coupling.

## 7. idpbuilder alternatives/successors

- **idpbuilder v0.10.2** (2026-04-13, Apache-2.0, active into July 2026) — kind + Gitea(+OCI registry) + Argo CD + ingress-nginx, manifests baked into binary, `-p` package stacks ([cnoe-io/idpbuilder](https://github.com/cnoe-io/idpbuilder)). Weaknesses to improve on: single cluster provider, Argo-only, YAML-only, no plugin system.
- **ksail** ([devantler-tech/ksail](https://github.com/devantler-tech/ksail), Go) — Kind/K3d/Talos/VCluster/KWOK/EKS × Flux/ArgoCD × SOPS/age, MCP server; the clearest "next-gen idpbuilder" blueprint.
- **Kratix** (Syntasso, Apache-2.0, active; enterprise SKE tier) — in-cluster Promise-based platform API orchestration; complementary (a stack to install), not a competitor.
- **vCluster Platform** — commercial control plane atop Apache-2.0 core.
- **Humanitec/Score** — Score is CNCF Sandbox (donated July 2024, vendor-neutral); useful as a workload-spec input format.

**Positioning:** the open niche is idpbuilder's one-command UX + ksail's provider/engine matrix + Helm-4-style Wasm/OCI extensibility + CUE-native typed config.

## extensibility-patterns

**Summary:** Best-in-class CLIs converge on four extensibility tiers: schema-versioned declarative config (k3d, Devbox), data-only packages distributed as OCI artifacts or git refs (Crossplane xpkg, Devbox plugins, Porter/CNAB), exec plugins with a git-based index (kubectl/krew, Helm 3), and typed gRPC subprocess plugins only where deep integration is needed (Terraform via hashicorp/go-plugin, Crossplane functions, Dagger). Notably, the ecosystem is moving away from both raw subprocess plugins (Helm 4 replaces them with Extism/Wasm for sandboxing) and heavyweight bundle specs (CNAB has stagnated). For a Go CLI assembling local Kubernetes developer platforms, the recommended stack is: declarative apiVersion/kind config as the core, OCI-distributed declarative packages plus embedded Helm SDK for content, krew-style exec plugins for new commands, and go-plugin gRPC reserved for custom provisioner backends.

**Key findings:**
- Terraform providers are separate binaries launched by the CLI that print a handshake line (CORE-PROTOCOL-VERSION|APP-PROTOCOL-VERSION|NETWORK-TYPE|ADDR|PROTOCOL) on stdout, then serve gRPC on loopback via hashicorp/go-plugin; go-plugin's VersionedPlugins map lets client and server negotiate a mutually supported protocol major version at handshake time.
- Terraform's plugin protocol is versioned protobuf (v5 and v6; v6 adds nested attributes, resource identity, ephemeral resources); Terraform core is protocol-agnostic via per-version GRPCProvider implementations, and the registry uses declared protocol versions as compatibility metadata when selecting plugin releases -- but providers get zero sandboxing and the SDKv2-to-plugin-framework migration shows high SDK-churn cost.
- kubectl plugins are just executables named kubectl-<name> on PATH (pure exec model, argv+env contract only); krew is a plugin manager whose index is a plain git repo of YAML manifests pointing at platform-specific sha256-pinned archives installed under ~/.krew/store.
- krew explicitly warns on every install that plugins 'are not audited for security by the Krew maintainers'; a Trend Micro analysis found issues in nearly all 154 scanned plugins -- the exec model's DX simplicity (any language, 5-minute start) comes with zero sandboxing and no version-compat contract with the host CLI.
- Helm 3 plugins are subprocess-based (plugin.yaml + command, incl. 'downloader' plugins keyed by URL scheme), but Helm 4 (HIP-0026, targeted for Helm v4.0.0 Nov 2025) replaces this with an Extism-based Wasm plugin system: typed plugin kinds (cli/v1, getter/v1, postrenderer/v1), protobuf/JSON message passing, mandatory signing with verification on by default, and Wasm sandboxing; legacy subprocess plugins are deprecated and slated for removal in Helm 5.
- The Helm Go SDK (helm.sh/helm/v3, now v4) is fully embeddable -- a platform-builder CLI can drive chart install/upgrade in-process via the action package instead of shelling out to the helm binary, which is the pattern HIP-0026 itself recommends for SDK users.
- Crossplane xpkg packages are plain OCI images (a superset of the OCI image spec) containing a package.yaml stream with optional io.crossplane.xpkg layer annotations, pushed to any container registry with semver dependency resolution -- a proven 'reuse OCI registries instead of building your own' distribution strategy.
- Crossplane composition functions are OCI-distributed gRPC servers implementing FunctionRunnerService (RunFunctionRequest/RunFunctionResponse carrying full observed+desired state); transport security is mandatory unless --insecure is set -- a clean typed-pipeline contract, but the full-desired-state-every-call semantics is a documented footgun (unspecified resources get deleted).
- Dagger is container-based, not Wasm: functions run inside containers spawned by the BuildKit-based Dagger Engine, exposed through a GraphQL universal type system; SDKs exist for Go, Python, TypeScript, PHP, Java, .NET, Elixir, Rust, and each SDK code-generates native bindings so a TypeScript module can natively call a Go module.
- Dagger modules are distributed as git references (GITSERVER/USER/REPO[/SUBPATH]@vX.Y.Z semver tags) recorded with resolved commit pins in dagger.json; Daggerverse is only an index/browser, not a host -- strong sandboxing and caching, but it drags in a running engine daemon and Docker dependency, which is heavy for a CLI.
- Porter extends via 'mixins' -- subprocess executables invoked with build/install/upgrade subcommands whose output is composed into a single CNAB invocation image Dockerfile at build time; bundles (bundle.json + invocation image) are pushed to OCI registries per CNAB Core 1.1.0 -- architecturally interesting but the CNAB ecosystem has largely stagnated and is a cautionary tale about spec-first extensibility.
- k3d has no plugin system at all: a schema-versioned config file (apiVersion: k3d.io/v1alpha5, kind: Simple) validated by a published JSON Schema, with 1:1 CLI-flag mapping and automatic migration between config versions -- minimal but extremely low-friction, and its multi-year 'v1alphaN' churn shows the cost of never stabilizing the schema.
- Devbox plugins are declarative data, not code: a plugin.json (Go JSON template) declaring packages, env, create_files (including process-compose.yaml services), and init_hook, includable from any git repo via 'github:<org>/<repo>?dir=<plugin-dir>' -- best-in-class authoring DX because most extensions need no logic at all; Flox is the same idea with manifest.toml over a Nix catalog.
- Ranked recommendation for the Go platform-builder CLI: (1) k3d-style versioned declarative config as the core contract; (2) Devbox-style data-only packages distributed as OCI artifacts (xpkg pattern via ORAS) + embedded Helm SDK; (3) krew-style exec plugins ('<cli>-<name>' on PATH + git-repo index with sha256 pins and an explicit trust warning) for new subcommands; (4) hashicorp/go-plugin gRPC with VersionedPlugins negotiation only for custom cluster-provisioner/engine backends; (5) track Extism/Wasm (Helm 4's bet) for future sandboxed logic hooks; avoid CNAB and avoid requiring a Dagger-style engine daemon.

## 1. Terraform / OpenTofu — go-plugin gRPC providers

Providers are standalone binaries launched by the CLI. Discovery/launch follows [hashicorp/go-plugin](https://pkg.go.dev/github.com/hashicorp/go-plugin): the child prints `CORE-PROTOCOL-VERSION|APP-PROTOCOL-VERSION|NETWORK-TYPE|NETWORK-ADDR|PROTOCOL` on stdout, the host connects as a gRPC client over loopback/unix socket. A "magic cookie" env var is an intentional UX check, not security. `VersionedPlugins` enables handshake-time negotiation of a mutually supported protocol major version. The wire contract is versioned protobuf ([plugin protocol v5/v6](https://developer.hashicorp.com/terraform/plugin/terraform-plugin-protocol); v6 adds nested attributes, resource identity, ephemeral resources), and core stays protocol-agnostic behind a single `providers.Interface`. Distribution: registry protocol (registry.terraform.io / OpenTofu registry) serving GPG-signed, per-platform zips, with protocol versions in the [release manifest](https://github.com/hashicorp/terraform/blob/main/docs/plugin-protocol/README.md) used for selection. Schema is dynamic: core calls GetProviderSchema and validates config against it.

Copy: handshake + version negotiation, schema-driven validation, protocol-agnostic core. Avoid: zero sandboxing (full user privileges), high extension-authoring cost (CRUD + schema + test harness), SDK churn (SDKv2 → plugin-framework, protocol mux shims).

## 2. kubectl / krew — exec plugins

Any executable named `kubectl-<name>` on PATH is a plugin; the contract is only argv + environment ([kubectl docs](https://kubernetes.io/docs/tasks/extend-kubectl/kubectl-plugins/)). [Krew](https://github.com/kubernetes-sigs/krew) adds distribution: the index is a git repo of YAML manifests pointing to platform-specific archives with sha256 digests, installed under `~/.krew/store`. Versioning between host and plugin is nonexistent; sandboxing is nonexistent — krew prints "not audited for security... run at your own risk" on every install, and [Trend Micro's scan](https://www.trendmicro.com/vinfo/us/security/news/cybercrime-and-digital-threats/protecting-your-krew-a-security-analysis-of-kubectl-plug-ins) found issues in ~153 of 154 plugins. DX is the best of any model: any language, ship in minutes. Failure modes: PATH collisions, no capability negotiation, no shared state with the host.

## 3. Helm — subprocess plugins today, Wasm tomorrow; embeddable SDK

Helm 3 plugins are `plugin.yaml` + subprocess, including scheme-keyed "downloader" plugins ([plugins guide](https://helm.sh/docs/topics/plugins/)). The decisive datapoint: [HIP-0026](https://helm.sh/community/hips/hip-0026/) replaces this in Helm 4 with an **Extism-based Wasm** system — typed plugin kinds (`cli/v1`, `getter/v1`, `postrenderer/v1`), protobuf/JSON message passing, mandatory signing verified by default, Wasm sandboxing; legacy subprocess is deprecated, removed in Helm 5. Separately, the Helm Go SDK (`helm.sh/helm/v3`/v4 `action` package) is designed to be embedded — a platform CLI should drive installs in-process rather than shelling to `helm`.

## 4. Crossplane — OCI xpkg + gRPC functions

[xpkg](https://github.com/crossplane/crossplane/blob/main/contributing/specifications/xpkg.md) is a plain OCI image (superset of the OCI image spec) carrying a `package.yaml` stream, with `io.crossplane.xpkg` layer annotations; pushed to any registry, with semver dependency resolution by the package manager. [Composition functions](https://github.com/crossplane/crossplane/blob/main/contributing/specifications/functions.md) are OCI-distributed gRPC servers (`FunctionRunnerService`, `RunFunctionRequest/Response` over full observed+desired state; TLS mandatory unless `--insecure`). Copy: piggyback on OCI registries (via ORAS) instead of running your own. Avoid: full-desired-state semantics (forgetting a resource deletes it) and revision-management complexity; the whole model presumes a running cluster.

## 5. Dagger — container functions behind GraphQL (not Wasm)

Correction to the brief: Dagger is **container-based**, not Wasm. Functions execute in containers on the BuildKit-based engine; the [GraphQL API](https://docs.dagger.io/api/reference/) is the universal, introspectable type system; SDKs (Go, Python, TS, PHP, Java, .NET, Elixir, Rust) code-generate native bindings so modules call each other across languages ([Dagger Functions](https://dagger.io/blog/introducing-dagger-functions/)). Modules are distributed as git refs `GITSERVER/USER/REPO[/SUBPATH]@vX.Y.Z` with resolved commit pins in `dagger.json`; [Daggerverse](https://docs.dagger.io/extending/daggerverse/) is an index, not a host. Copy: git-ref + semver + commit-pin distribution, cross-language codegen idea, container sandboxing. Avoid: requiring an engine daemon + Docker — too heavy as the baseline for a CLI.

## 6. Porter / CNAB

Bundles = `bundle.json` + an invocation image, pushed to OCI registries (CNAB Core 1.1.0, [porter.sh/cnab](https://porter.sh/cnab/)). Extensibility is via **mixins**: subprocess executables Porter invokes with `build`/`install`/etc.; at build time each mixin's `build` output is appended into the invocation-image Dockerfile. Elegant composition, but the CNAB ecosystem has stagnated since ~2021 — a cautionary tale about betting on a heavyweight spec before ecosystem pull exists.

## 7. k3d — config-file-only

No plugin system. `apiVersion: k3d.io/v1alpha5, kind: Simple` config validated by a published [JSON Schema](https://k3d.io/v5.1.0/usage/configfile/), 1:1 flag mapping, automatic migration between config versions. Copy: schema-versioned config with published JSON Schema (editor autocomplete, lintability). Avoid: staying "v1alphaN" for years; extensibility ceiling is whatever core supports.

## 8. Devbox / Flox — declarative environments with data-only plugins

`devbox.json` declares packages; [Devbox plugins](https://www.jetify.com/docs/devbox/guides/plugins) are pure data (`plugin.json`: `packages`, `env`, `create_files` — including `process-compose.yaml` services — `init_hook`, `include`), loadable from any git repo via `github:<org>/<repo>?dir=<dir>`. Nix supplies reproducibility. This is the best authoring DX observed: most "extensions" in an assembly tool are configuration + files + services, requiring no code and thus minimal sandboxing surface (only `init_hook` runs shell).

## Ranked recommendation (Go CLI assembling local K8s dev platforms)

1. **Core: versioned declarative config** (k3d/Devbox pattern). `apiVersion`/`kind` platform spec, published JSON Schema, explicit config-version migration. This is the product, not a plugin mechanism.
2. **Primary extension: data-only packages over OCI**. Devbox-style declarative packages (manifests, Helm chart refs, values, service defs, env, hooks) packaged as OCI artifacts via ORAS (xpkg lesson), also loadable from git refs with commit pins (Dagger lesson). Embed the Helm SDK (`action` package) in-process. This covers ~80% of real extensions with near-zero authoring cost and near-zero attack surface.
3. **Command-surface extension: krew-style exec plugins**. `<cli>-<name>` binaries on PATH, a git-repo index of YAML manifests with sha256-pinned per-platform archives, explicit unaudited-plugin warning. Cheap, language-agnostic; accept the trust tradeoff openly.
4. **Deep integration: hashicorp/go-plugin gRPC**, only for pluggable provisioner/engine backends (e.g., alternative cluster runtimes) where you need long-lived typed bidirectional calls. Use `VersionedPlugins` negotiation and a versioned protobuf contract from day one; keep core behind a protocol-agnostic interface like Terraform does.
5. **Future: Extism/Wasm hooks** for sandboxed logic (validators, transformers), following Helm 4's direction — adopt once Helm 4's system proves out, not before.

**Avoid:** CNAB/Porter-style bundle specs (ecosystem risk), requiring a Dagger-style engine daemon, running a bespoke plugin registry (use OCI + git), and unversioned subprocess contracts (Helm 3's regret).
